# Infrastructure

CrestPet System runs on AWS with infrastructure managed by Terraform. The compute model differs by environment:

- **Prestaging:** EKS multi-region active-active stack live, with Singapore (`ap-southeast-1`) as the primary region and `us-east-1` as the secondary. Aurora Global Database and Redis Global Datastore replicate across both regions.
- **Staging:** Same EKS multi-region active-active topology as prestaging (Singapore primary + `us-east-1` secondary, Aurora Global + Redis Global).
- **Production:** Still single-region ECS Fargate in `us-east-1`, fronted by RDS PostgreSQL, ElastiCache Redis, ALB, WAF, and CloudFront. The multi-region cutover is not yet scheduled.

For the operator's guide to the multi-region stack (failover runbooks, replication health, region-specific endpoints), see [Multi-Region Architecture](/operations/multi-region-architecture).

## AWS Services

| Service | Purpose | Details |
|---------|---------|---------|
| **ECS Fargate** | Compute (Primary) | Serverless containers for Next.js frontend and NestJS backend |
| **EKS** | Compute (Provisioned, not active) | Kubernetes module exists in Terraform for future multi-region active-active deployment, but is not currently serving production traffic |
| **RDS PostgreSQL** | Database | Managed PostgreSQL with automated backups and encryption |
| **ElastiCache Redis** | Cache/Queue | Session storage, caching layer, BullMQ job queues, Socket.IO adapter |
| **S3** | File Storage | Documents, images, scans, exports, uploaded files |
| **CloudFront** | CDN | Static asset delivery, SSL termination |
| **ALB** | Load Balancer | Application Load Balancer with path-based routing |
| **WAF** | Security | Web Application Firewall with managed rule groups |
| **Secrets Manager** | Secrets | API keys, database credentials, third-party tokens |
| **SES** | Email | Transactional email delivery |
| **CloudWatch** | Monitoring | Logs, metrics, alarms |

## Three Environments

CrestPet runs across three environments, each in AWS `us-east-1`:

| Environment | Purpose | Deployment |
|-------------|---------|------------|
| **Prestaging** | Active development and feature testing. Developers deploy here for integration testing. | Automatic on push to `prestaging` branch |
| **Staging** | Pre-production validation. Mirrors production configuration for final QA. | Manual promotion from prestaging |
| **Production** | Live customer-facing environment. | Manual promotion from staging (never from CLI) |

<Warning>
  **Never run `terraform apply` for production from the CLI.** Production deployments go through the CI/CD pipeline only.
</Warning>

## Deployment Pipeline

```
Developer → Push to prestaging branch
         → GitHub Actions CI
            ├── Lint + Type Check
            ├── Unit Tests
            ├── E2E Tests (test containers)
            ├── SonarCloud Analysis
            ├── Build Docker Images
            └── Deploy to ECS (prestaging)

Promotion → Staging (manual trigger)
         → Production (manual trigger, requires approval)
```

## Networking

- **Subdomain routing**: Each portal is served from a different subdomain (e.g., `clinic.crestpet.com`, `consumer.crestpet.com`)
- **ALB routing**: Path-based rules route `/api/*` to the NestJS backend and all other paths to the Next.js frontend
- **VPC**: All services run within a private VPC with NAT gateways for outbound internet access
- **Security Groups**: Strict ingress/egress rules between services

## Security

- **WAF** protects the ALB with managed rule groups (SQL injection, XSS, known bad inputs)
- **Secrets Manager** stores all credentials -- no plaintext secrets in code or environment variables
- **Encryption at rest** on RDS, S3, and EBS volumes
- **Encryption in transit** via TLS everywhere (CloudFront, ALB, database connections)
- **Turnstile** (Cloudflare) for bot protection on public-facing forms
- **Secret scanning** and **push protection** enabled via GitHub
- **Dependabot** for automated dependency vulnerability alerts

## Infrastructure as Code

The production infrastructure is managed in a separate Terraform repository ([`cmistaff/infra`](https://github.com/cmistaff/infra)). Key modules:

- ECS cluster and service definitions
- RDS instance configuration
- ElastiCache cluster
- ALB listeners and target groups
- CloudFront distributions
- S3 buckets and policies
- WAF web ACLs
- IAM roles and policies
- VPC, subnets, and security groups

## Development Infrastructure

Developer workstations are EC2 instances managed through a separate Terraform repository ([`cmistaff/EC2DevsCrestPet`](https://github.com/cmistaff/EC2DevsCrestPet)):

- 22 EC2 boxes (21 running, 1 stopped) across three regions:
  - **Ohio (`us-east-2`):** 3 instances -- `dev-dylan` (m6i.4xlarge, stopped), `dev-bryant` (m6i.12xlarge, running), `dev-michael` (m6i.12xlarge, running)
  - **Mumbai (`ap-south-1`):** 18 instances, all m6i.4xlarge
  - **Frankfurt (`eu-central-1`):** 1 instance -- `dev-serhii` (m6i.4xlarge)
- Connected via Tailscale VPN mesh network with custom DERP relays in Ohio (`derp.crestbrain.com`, region 900) and Mumbai (`derp-mumbai.crestbrain.com`, region 901). Frankfurt routes via the Mumbai relay.
- noVNC for browser-based desktop access
- Standardized AMIs built with Packer
- Memory watchdog to prevent OOM issues
- CloudWatch agent for CPU, network, and TCP metrics

Self-hosted GitHub Actions runners (managed via [`cmistaff/github-runners-infrastructure`](https://github.com/cmistaff/github-runners-infrastructure)) provide CI/CD capacity with Spot and on-demand ASGs.
