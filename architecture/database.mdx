# Database Architecture

CrestPet uses PostgreSQL as its primary database, accessed through Prisma 6 ORM. The schema is split across 90+ files totaling roughly 12,000 lines and ~325 models, organized by feature domain.

## Multi-Tenancy Model

CrestPet uses a **shared database, shared schema** multi-tenancy model. All tenants share the same PostgreSQL database and tables, with row-level isolation enforced through `tenantId` columns.

### Tenant Model

```prisma
model Tenant {
  id                  String       @id @default(uuid())
  slug                String       // Subdomain identifier (e.g., "mars")
  name                String       // Display name (e.g., "Mars Veterinary Clinic")
  type                TenantType   // CLINIC, ENTERPRISE, CONSUMER, PLATFORM, REFERRAL_PORTAL
  domain              String?
  timezone            String       @default("UTC")
  measurementSystem   String       @default("imperial")
  status              TenantStatus @default(ACTIVE)  // ACTIVE, INACTIVE, SUSPENDED
  ownedByEnterpriseId String?      // Enterprise ownership link

  // ... 100+ relationships to domain models
}
```

### Tenant Types

| Type | Purpose |
|------|---------|
| `CLINIC` | Individual veterinary practice |
| `ENTERPRISE` | Corporate entity owning multiple clinics |
| `CONSUMER` | Pet owner tenant (auto-created) |
| `PLATFORM` | System-wide admin tenant |
| `REFERRAL_PORTAL` | Referral network tenant |

### Enterprise-Clinic Ownership

Enterprises own clinics through a self-referential relationship:

```prisma
model Tenant {
  ownedByEnterprise Tenant?  @relation("EnterpriseOwnership", fields: [ownedByEnterpriseId], references: [id])
  ownedClinics      Tenant[] @relation("EnterpriseOwnership")
}
```

### User-Tenant Association

Users are linked to tenants through `TenantUser`, which also carries role assignments:

```prisma
model TenantUser {
  id           String     @id @default(uuid())
  userId       String
  tenantId     String
  tenantRoleId String
  isActive     Boolean    @default(true)
  joinedAt     DateTime   @default(now())

  user   User       @relation(...)
  tenant Tenant     @relation(...)
  role   TenantRole @relation(...)

  @@unique([userId, tenantId])
}
```

A single user can belong to multiple tenants with different roles.

## Authentication Schema

The auth schema follows Better Auth conventions with Prisma adapter support:

| Model | Purpose |
|-------|---------|
| `Session` | Active user sessions with token, expiry, IP, and user agent |
| `Account` | OAuth provider accounts (credentials, Google, etc.) |
| `Verification` | Email/phone verification tokens |
| `Passkey` | WebAuthn/FIDO2 credentials for passwordless auth |
| `OtpCode` | Phone-based one-time passwords |
| `EmailOtpCode` | Email-based one-time passwords |
| `PasswordResetToken` | Password reset flow tokens |
| `AccountReactivationToken` | Reactivation of soft-deleted accounts (90-day window) |
| `EmailChangeRequest` | Two-factor email change verification (SMS + email) |

## Role-Based Access Control

Permissions are managed through dedicated tables with a hierarchical Permission enum and CRUD boolean flags:

```prisma
model TenantRole {
  id              String                 @id @default(uuid())
  tenantId        String
  name            String
  description     String?
  color           String?                @default("#6366f1")
  rolePermissions TenantRolePermission[]
  tenantUser      TenantUser[]
  createdAt       DateTime               @default(now())
  updatedAt       DateTime               @updatedAt
}

model TenantRolePermission {
  id           String     @id @default(uuid())
  tenantId     String
  tenantRoleId String
  permission   Permission // Hierarchical enum (e.g., CLINIC__APPOINTMENTS, CLINIC__PAYMENTS)
  canCreate    Boolean    @default(false)
  canRead      Boolean    @default(false)
  canUpdate    Boolean    @default(false)
  canDelete    Boolean    @default(false)
}
```

The `Permission` enum contains ~70 hierarchical values using double-underscore naming (e.g., `CLINIC__APPOINTMENTS`, `CLINIC__PAYMENTS`, `CLINIC__INVENTORY`). Each permission has four independent boolean flags for granular CRUD control rather than a single access level.

## Tenant Settings (Dedicated Tables)

Tenant configuration has been migrated from a monolithic JSON `settings` field to dedicated tables (11 total):

| Table | Configuration Area |
|-------|-------------------|
| `TenantBrandingSettings` | Logo, colors, custom CSS |
| `TenantOrganizationSettings` | Business hours, contact info |
| `TenantSecuritySettings` | Password policies, session timeouts |
| `TenantNotificationSettings` | Notification preferences |
| `TenantIntegrationSettings` | Third-party integrations |
| `TenantPaymentSettings` | Payment configuration |
| `TenantPortalSettings` | Portal-specific settings |
| `TenantBackupSettings` | Backup and data retention configuration |
| `TenantCreditSettings` | Tenant-level prepaid credit balance and clinic credit configuration |
| `TenantVideoConsultationProviderSettings` | Per-tenant video consultation provider routing (LiveKit / Daily) |
| `TenantMiscSettings` | Miscellaneous settings (JSON value field) |

The legacy `settings` JSON field is retained for backward compatibility but new features must use dedicated tables.

## Key Conventions

- **All dates stored in UTC.** Timezone conversion happens at the application layer.
- **Soft deletes** via `deletedAt` fields where needed.
- **UUID primary keys** on all models (`@default(uuid())`).
- **Explicit `select` in all queries** -- never `include`, never bare `findMany()`.
- **Indexes** on `tenantId`, `slug`, `type`, `deletedAt`, and frequently queried foreign keys.

## Migrations

Migrations are managed through Prisma:

```bash
pnpm prisma:migration:create <name>   # Create a new migration
pnpm prisma:migration:run             # Apply pending migrations
pnpm prisma:migration:reset           # Reset and re-apply all migrations
pnpm prisma:generate                  # Regenerate Prisma client + server repositories
```

Migration files are never modified manually. The generated SQL is treated as immutable.
