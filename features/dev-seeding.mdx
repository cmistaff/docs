## Problem Statement [#problem-statement]

The `seed.ts` script is currently not committed to version control because running it in production would be dangerous (could overwrite production data with seed data). However, this creates friction for developers who need to:

- Share seed script updates with the team
- Keep seed data in sync as features get merged
- Work with the complex Postgres schema without manual DB editing tools

## Proposed Solution [#proposed-solution]

Implement an automated dev database seeding pipeline inspired by the Knight Hacks project setup.

### Architecture Overview [#architecture-overview]

```
Prestaging DB
 ↓
 [Clone via pg_dump]
 ↓
 Prestaging Clone (temp CI database)
 ↓
 [Run seed.ts]
 ↓
 Seeded Dev DB Dump
 ↓
 [Upload to S3]
 ↓
 Developers run `pnpm db:fetch`, which retrieves dump
 from S3 and imports to local dev database.
```

> **Note:** Prestaging contains only development/test data - no real customer PII or production data. The database can be cloned as-is without sanitization.

## Automation Flow [#automation-flow]

**On merge to prestaging:**

1. GitHub Actions workflow triggers
2. Spin up temp PostgreSQL container, run Prisma migrations
3. Clone prestaging data via `pg_dump` from read-replica
4. Import clone into temp container
5. Run `seed.ts` against the cloned data
6. Dump the seeded database, compress, and upload to S3

### Architecture Diagram [#architecture-diagram]

```
┌─────────────────────────────────────────────────────────────────┐
│ GitHub Actions (on merge to prestaging) │
├─────────────────────────────────────────────────────────────────┤
│ 1. Spin up temp PostgreSQL container │
│ 2. Run Prisma migrations to create schema │
│ 3. pg_dump prestaging read-replica → import into temp DB │

│ 5. pg_dump → gzip → upload to S3 │
└─────────────────────────────────────────────────────────────────┘
 ↓
┌─────────────────────────────────────────────────────────────────┐
│ S3 Bucket (crest-pet-dev-db-dumps) │
│ └── dumps/seeded-dev-db.sql.gz (latest) │
│ └── dumps/archive/seeded-dev-db-{timestamp}.sql.gz │
└─────────────────────────────────────────────────────────────────┘
 ↓
┌─────────────────────────────────────────────────────────────────┐
│ Developer: pnpm db:fetch │
├─────────────────────────────────────────────────────────────────┤
│ 1. Validate NODE_ENV=development & DATABASE_URL not production │
│ 2. Download seeded-dev-db.sql.gz from S3 │
│ 3. Validate SHA256 checksum │
│ 4. Extract and import using db:reset-and-import pattern │
└─────────────────────────────────────────────────────────────────┘
```

## Safety Guards [#safety-guards]

| Guard | Implementation |
| --- | --- |
| NODE_ENV check | db:fetch validates development or test only |
| Database URL validation | Rejects URLs containing production, crest.vet, crest.pet, rds.amazonaws.com |
| Source database safety | Pipeline only connects to prestaging read-replica |
| Dump integrity | SHA256 checksum generated and validated on download |

## Blockers & Prerequisites [#blockers-prerequisites]

### 1. S3 Bucket Creation (CRITICAL) [#1-s3-bucket-creation-critical]

**Bucket Name:** `crest-pet-dev-db-dumps` **Region:** `us-east-1`

**Required Configuration:**

- **Encryption:** AES-256 (SSE-S3)
- **Versioning:** Enabled
- **Lifecycle Rules:**Delete non-current versions after 30 days
- Delete incomplete multipart uploads after 7 days


**Public Access:** Block all public access

**Verification:**

bash

```
aws s3 ls s3://crest-pet-dev-db-dumps/
# Should return empty (no error)
```

### 2. IAM Policy Updates (CRITICAL) [#2-iam-policy-updates-critical]

**Role:** `crest-dev-ec2-role` (and GitHub Actions OIDC role)

**Required Permissions:**

json

```
{
 "Version": "2012-10-17",
 "Statement": [
 {
 "Sid": "DevDbDumpsAccess",
 "Effect": "Allow",
 "Action": [
 "s3:GetObject",
 "s3:PutObject",
 "s3:ListBucket",
 "s3:DeleteObject"
 ],
 "Resource": [
 "arn:aws:s3:::crest-pet-dev-db-dumps",
 "arn:aws:s3:::crest-pet-dev-db-dumps/*"
 ]
 }
 ]
}
```

**Roles that need this policy:**

| Role | Purpose |
| --- | --- |
| crest-dev-ec2-role | Developer VMs to run pnpm db:fetch |
| GitHub Actions OIDC Role | CI pipeline to upload dumps |

### 3. GitHub Secret (CRITICAL) [#3-github-secret-critical]

**Environment:** `prestaging` **Secret Name:** `PRESTAGING_READ_REPLICA_URL`

**Value Format:**

```
postgresql://username:password@prestaging-read-replica-host:5432/crest_pet?schema=public
```

### 4. Environment Variable (LOW PRIORITY) [#4-environment-variable-low-priority]

**File:** `server/.env` (local development)

```
AWS_S3_BUCKET_DEV_DB_DUMPS=crest-pet-dev-db-dumps
```

> **Note:** Only needed if services reference the bucket programmatically. The `db:fetch` script hardcodes the bucket name.

## Dev DB Seeding Pipeline - Infra Requirements [#dev-db-seeding-pipeline-infra-requirements]

 Automated pipeline creates seeded dev DB dumps on push to prestaging. Devs consume via pnpm db:fetch.

### AWS Resources (Terraform) [#aws-resources-terraform]

- S3 Bucket: crest-pet-dev-db-dumps (us-east-2, SSE-AES256). Already created manually by Michael - needs terraform import.
- S3 Objects: dumps/seeded-dev-db.sql.gz (latest), .sha256 (checksum), dumps/archive/* (timestamped)
- IAM Role (pipeline): OIDC-based, stored as SEED_AWS_ROLE_TO_ASSUME secret. Needs s3:PutObject + s3:ListBucket on bucket, and rds-db:connect for viewer_iam DB user.
- IAM Policy (devs): Devs use own AWS creds. Need s3:GetObject on dumps/*

### GitHub Secrets [#github-secrets]

- SEED_AWS_ROLE_TO_ASSUME - New IAM role ARN from Terraform output (Infra team)
- PRESTAGING_DB_HOST - RDS endpoint hostname (Infra team)
- PRESTAGING_DB_NAME - Database name: crest_pet_prestaging (Infra team)
- SLACK_WEBHOOK_URL - Already exists

### IAM Role Permissions for SEED_AWS_ROLE_TO_ASSUME [#iam-role-permissions-for-seedawsroletoassume]

- s3:PutObject on arn:aws:s3:::crest-pet-dev-db-dumps/dumps/*
- s3:GetObject on arn:aws:s3:::crest-pet-dev-db-dumps/dumps/*
- s3:ListBucket on arn:aws:s3:::crest-pet-dev-db-dumps
- rds-db:connect on arn:aws:rds-db:{region}:{account}:dbuser:{dbi-resource-id}/viewer_iam

### OIDC Trust Policy [#oidc-trust-policy]

 Trust: [token.actions.githubusercontent.com](http://token.actions.githubusercontent.com)
 Audience: [sts.amazonaws.com](http://sts.amazonaws.com)
 Subject: repo:cmistaff/crestpetsystem:*

### Runner Requirements [#runner-requirements]

 Self-hosted [build] runner with: Docker, pg_dump/psql, gzip, sha256sum, aws CLI, Node 22.x + pnpm.
 Network access to prestaging RDS, S3, and GitHub OIDC endpoint.

### DB Access Method [#db-access-method]

 Uses IAM database authentication with the existing viewer_iam role (SELECT-only). No read replica needed.
 Workflow generates a 15-minute auth token via aws rds generate-db-auth-token at runtime. Requires SSL
 (PGSSLMODE=require).
