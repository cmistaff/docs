## Executive Summary

- **Goal**: Move Marcia AI execution to **Marcia Omnichannel MCP** (Chat + SMS + Voice + WhatsApp + Email), while **CrestPetSystem** remains the product UI + source of truth for business data/actions.
- **Current status (grounded)**:**MCP**: Orchestrator + tool registry + [http://Socket.IO](http://Socket.IO)  chat are implemented; SMS + voice handlers exist but **Express routes are still stubs** (`marcia-omnichannel-mcp/src/routes/channels.ts:12-52`). WhatsApp + Email are not implemented yet (see B7/B8).
- **Infra**: Terraform provisions **SQS + a queue-consumer ECS service** (`infra/modules/mcp/sqs.tf`), but MCP does not currently ship a working queue consumer (and the MCP `Dockerfile` attempts to build a non-existent `src/queue/consumer.ts` ,  `marcia-omnichannel-mcp/Dockerfile:45-50`).
- **CrestPetSystem integration**: MCP tools rely on `CrestDataService`, which currently calls `/api/...` endpoints that do not exist in CrestPetSystem as implemented today (see “Blockers” below).

### Production blockers (must-fix to ship)

- **B1 ,  Crest API contract mismatch (largest blocker)**:MCP tools call `CrestDataService` methods (29 tool files) which call paths like `/api/users/:id/...` (`marcia-omnichannel-mcp/src/data/crest-data-service.ts:77-347`).
- CrestPetSystem controllers mount at **root paths** via `Controller(name)` (`crestpetsystem/server/src/openapi/decorators/api-controller.decorator.ts:20-21`), and the frontend calls paths like `/platform/users` (example: `crestpetsystem/client/src/app/platform/dashboard/users/components/WaitingListTable.tsx:52-55`).
- **Result**: Without either (a) adding a Crest “MCP Internal API” matching MCP’s contract, or (b) changing MCP’s contract + tools, the MCP tool layer will fail in prod.
**B2 ,  Voice webhook routing is path-mismatched**:

- MCP voice handler is explicitly designed for **two endpoints**: `/voice/initiation` and `/voice/tool` (`marcia-omnichannel-mcp/src/channels/voice/handler.ts:5-9`).
- Current router only exposes `/voice/webhook` and `/elevenlabs/webhook` stubs (`marcia-omnichannel-mcp/src/routes/channels.ts:21-52`).

**B3 ,  MCP chat JWT secret selection will break Crest tokens**:

- Socket auth uses `deps.config.BETTER_AUTH_SECRET ?? deps.config.JWT_SECRET` (`marcia-omnichannel-mcp/src/websocket.ts:103-105`).
- In prod we will set `BETTER_AUTH_SECRET` (for service tokens), so MCP will try to verify Crest user JWTs with the wrong secret and reject all Crest chat connections.

**B4 ,  SQS/queue-consumer is half-wired (infra + Dockerfile expect it; MCP app doesn’t ship it)**:

- Infra provisions SQS + DLQ and a queue consumer service configured to run `bun run dist/queue/consumer.js` (`infra/modules/mcp/sqs.tf:12-138`).
- MCP repo does not currently include `src/queue/consumer.ts` (but the Dockerfile tries to build it ,  `marcia-omnichannel-mcp/Dockerfile:45-50`), so a Docker build from this repo will fail and the queue-consumer task cannot start as configured.
- SQS is also referenced in `infra/modules/mcp/main.tf:71`, `infra/modules/mcp/secrets.tf:22`, `infra/modules/mcp/iam.tf:100-112`, `infra/modules/mcp/cloudwatch.tf:15-45`, `infra/modules/mcp/outputs.tf:26-98`, plus environment outputs and setup docs.
- **Decision required**: either (a) remove SQS/queue-consumer fully (Terraform + docs + Dockerfile + dependencies), or (b) implement and ship the queue consumer end-to-end.

**B5 ,  Chat cutover data plane mismatch (conversations/messages)**:

- Crest consumer Marcia UI currently reads/writes chat history via Crest OpenAPI endpoints (`crestpetsystem/client/src/app/consumer/dashboard/marcia/components/consumer-marcia-chat.tsx:20-25` → `chatbot/conversations` controller).
- MCP chat persists messages to MCP’s own DB and rehydrates via `conversation_snapshot` (`marcia-omnichannel-mcp/src/websocket.ts:475-518`).
- **Result**: If the browser socket moves to MCP but the UI still loads history from Crest, the conversation list/messages will drift or appear empty.

**B6 ,  Chat file uploads/attachments not supported in MCP yet**:

- Crest chat UI sends base64 files over [http://Socket.IO](http://Socket.IO)  (`crestpetsystem/client/src/lib/hooks/use-marcia-socket.ts:119-156`).
- MCP [http://Socket.IO](http://Socket.IO)  payload currently declares `files?: unknown[]; // Ignoring file uploads for now` (`marcia-omnichannel-mcp/src/websocket.ts:22-27`).
- **Result**: If you cut chat over to MCP without implementing file handling, attachments will silently not work (or will require UI disablement).

**B7 ,  WhatsApp channel not implemented (but config exists)**:

- Express route is stubbed only (`marcia-omnichannel-mcp/src/routes/channels.ts:32-41`).
- MCP config already anticipates WhatsApp secrets/tokens (`marcia-omnichannel-mcp/src/services/config.ts:140-146`), but there is **no **`src/channels/whatsapp/` implementation yet.

**B8 ,  Email channel not implemented (but config exists for outbound)**:

- MCP channel enums include `email` (`marcia-omnichannel-mcp/src/http/schemas.ts:15`), but there is **no **`src/channels/email/` implementation yet.
- MCP config includes SendGrid keys (`marcia-omnichannel-mcp/src/services/config.ts:148-151`), but MCP `EnvSchema` (Zod) does not currently include SendGrid/WhatsApp-Meta keys (`marcia-omnichannel-mcp/src/config/env.ts`).
- **Result**: Implementing Email/WhatsApp cleanly requires aligning env validation + Terraform secret mappings to the actual variables used.

## Recommended rollout strategy (ASAP but safe)

**Phase 1 (ASAP)**: Cut over **Chat** to MCP behind a client-side switch (feature flag/env var). Deliver *minimum* Crest “MCP Internal API” endpoints required for tool-backed chat success.

**Phase 2**: Enable **SMS** (Twilio webhook → MCP), once service token + internal API are stable.

**Phase 3**: Enable **Voice** (ElevenLabs initiation + tool webhooks → MCP), after routing + raw-body signature validation are proven in prestaging.

**Phase 4**: Enable **WhatsApp** (MCP as webhook receiver + sender) with strict request verification.

**Phase 5**: Enable **Email** (MCP as inbound processor + sender) with strict request verification.

This reduces blast radius while unblocking MCP in production quickly.

## Production definition of done (DoD)

### Chat (Web)

**Client**: Crest frontend connects to MCP [http://Socket.IO](http://Socket.IO)  namespace `/marcia-chat` using a dedicated MCP base URL (not `commonConfig.api.baseUrl`).

**Auth**: MCP accepts Crest JWT access tokens and enforces `userId` match on `join_conversation`.

**Core loop**: `join_conversation` → `send_message` returns `message_response`, with `status_update` + `tool_execution` streaming.

**Persistence**: Conversation transcript writes succeed (or fail gracefully) and message history endpoint returns messages.

**Observability**: CloudWatch logs show traceable request/session IDs; errors are actionable.

**Rollback**: Flip feature flag/env var to return chat to Crest backend in < 5 minutes.

### SMS (Twilio)

**Inbound**: Twilio webhook hits MCP `/channels/sms/webhook` and is validated (signature required in staging/prestaging/production).

**Outbound**: MCP sends replies via Twilio API using `TWILIO_PHONE_NUMBER`.

**Safety**: Webhook handler is idempotent enough to tolerate retries; PII logging is minimized/redacted.

### Voice (ElevenLabs)

**Routing**: ElevenLabs initiation webhook and tool webhook are both wired and return 200 responses:

`/channels/voice/initiation` returns context

`/channels/voice/tool` executes tools and returns results

**Validation**: Signature validation is enabled in staging/prestaging/production and does not false-negative due to request body canonicalization.

### Infra

**Terraform**: `infra` applies cleanly with MCP enabled; **either** (a) SQS/queue-consumer resources are removed and no longer referenced, **or** (b) the queue consumer is implemented and the ECS queue-consumer service is healthy (no missing `dist/queue/consumer.js` failures).

**Secrets**: Required secrets exist in Secrets Manager and are mapped into the ECS task definition.

**Deploy**: ECS service is healthy behind ALB, `/health` is green, and logs/alarms are wired.

### WhatsApp

**Inbound**: WhatsApp provider webhook hits MCP and is **verified** (signature + verification token handshake).

**Outbound**: MCP can send replies via the selected provider (Meta Cloud API or Twilio) using configured credentials.

**Safety**: Rejects replay/invalid signatures; redacts PII; stores transcript entries with `channel: "whatsapp"`.

### Email

**Inbound**: Inbound email webhook/event is authenticated (recommended: AWS SES → internal Lambda → MCP with HMAC).

**Outbound**: MCP can send replies using configured sender (SendGrid or SES).

**Safety**: Rejects spoofed/untrusted payloads; stores transcript entries with `channel: "email"`.

**Production rollout** (canary + monitoring + rollback plan ready).

**Why**: fastest path to unblock 29 MCP tools without rewriting MCP.

**Decision 0.2 (required)**: Choose the **source of truth for chat history** during the first production rollout:

**Option A (ASAP MVP)**: MCP is realtime source-of-truth; Crest history UI becomes “best effort” (either disable history UI temporarily or rehydrate from MCP snapshots only).

**Option B (recommended for a normal production bar)**: Ensure Crest UI history stays correct by either:

(B1) making Crest UI consume MCP’s `conversation_snapshot` and stop calling Crest’s `chatbot/conversations/:id/messages`, or

(B2) having MCP also write chat messages to Crest (internal endpoint), until the UI fully migrates.

**Decision 0.3 (required)**: Decide whether MCP will use **SQS + a queue consumer** (and therefore must implement a queue consumer entrypoint), or stay **purely webhook-driven** (and remove SQS/queue-consumer infra + artifacts). Today, infra provisions SQS/consumer but MCP does not ship `src/queue/consumer.ts` (see B4).

#### Wiring pattern (important detail: per-request traceId)

Both `createSmsWebhookHandler` and `createVoiceToolHandler` want a `createContext` function, but you usually want the **requestId** as the traceId.

**Recommended pattern** in `marcia-omnichannel-mcp/src/routes/channels.ts`:

Inside each route, read `const requestId = req.requestId ?? crypto.randomUUID()`

Instantiate the handler **inside** the route so the closure captures `requestId` and passes it into `createToolContext(deps, requestId, sessionMeta)`

**Suggested wiring pattern (snippet):**

```
// marcia-omnichannel-mcp/src/routes/channels.ts
router.post("/sms/webhook", (req, res) => {
  const requestId = req.requestId ?? crypto.randomUUID();
  const handler = createSmsWebhookHandler({
    toolRegistry: deps.toolRegistry,
    createContext: (sessionMeta) => createToolContext(deps, requestId, sessionMeta),
  });

  return handler(req, res);
});
```

#### Signature validation detail (ElevenLabs raw-body risk)

ElevenLabs validation uses `timestamp + "." + body` (`marcia-omnichannel-mcp/src/channels/voice/elevenlabs-client.ts:135-139`), but the handler currently passes `JSON.stringify(req.body)` (`marcia-omnichannel-mcp/src/channels/voice/handler.ts:99-105,199-205`).

**Implementation requirement**:

Capture the **raw request body bytes** in Express middleware (for JSON) and validate against the raw body string.

If you don’t, you will likely get false negatives in staging/prod where signature validation is required.

**Suggested raw-body capture (snippet):**

```
// marcia-omnichannel-mcp/src/server.ts
type RawBodyRequest = express.Request & { rawBody?: string };

app.use(
  express.json({
    limit: "10mb",
    verify: (req: RawBodyRequest, _res, buf) => {
      // Attach rawBody for signature validation (ElevenLabs)
      req.rawBody = buf.toString("utf8");
    },
  }),
);
```

#### WhatsApp authentication (Meta Cloud API “perfect auth”)

MCP already has config placeholders for Meta WhatsApp (`WHATSAPP_PHONE_ID`, `WHATSAPP_ACCESS_TOKEN`, `WHATSAPP_VERIFY_TOKEN`, `WHATSAPP_APP_SECRET`) in `marcia-omnichannel-mcp/src/services/config.ts:142-146`, but the channel is not implemented.

**Verification handshake (GET)**:

Validate `hub.mode === "subscribe"`

Validate `hub.verify_token === WHATSAPP_VERIFY_TOKEN`

Respond with `hub.challenge` (plain text)

**Signature validation (POST)**:

Read `X-Hub-Signature-256`

Compute HMAC-SHA256 over the **raw request body** using `WHATSAPP_APP_SECRET`

Constant-time compare and reject invalid signatures

**Payload integrity checks**:

Validate payload contains the expected `phone_number_id` matching `WHATSAPP_PHONE_ID`

Reject if mismatch (prevents cross-app spoofing)

#### Email authentication (recommended “perfect auth” pattern)

Because email providers vary and many inbound webhooks are difficult to strongly authenticate, the recommended production pattern is:

**Inbound**: AWS SES → SNS/Lambda → MCP `POST /channels/email/inbound`

**Auth**: Lambda signs its request to MCP with an HMAC header (shared secret), including a timestamp and nonce to prevent replay.

**Reason**: This makes MCP trust only *our* AWS pipeline (IAM + internal signing), not random internet POSTs.

#### Tenant-scoping note (do not ignore)

Many existing Crest services require `tenantId` / tenant context (see e.g. consumer appointment controller requiring `@CurrentTenant` ,  `crestpetsystem/server/src/modules/portals/consumer/appointment/appointment.controller.ts:57-63`).

To ship quickly without re-architecting Crest:

**Recommended internal API behavior**:

For “user-owned” reads (pets, appointments, documents), allow **no explicit tenant input** and implement queries that only return entities where:

the entity is linked to the `userId`, and

the entity’s tenant is one where the user is active (via `TenantUser` join).

For “tenant-specific” operations, accept tenant identifiers explicitly (e.g. `tenantSlug` already appears in clinic routes).

This avoids relying on `DomainContextService` (which normally comes from tenant host headers) while still enforcing tenant isolation.

### IB-4: Chat cutover implementation details (B5/B6)

#### What Crest UI currently does

Creates and lists conversations via Crest OpenAPI:

`POST /chatbot/conversations`

`GET /chatbot/conversations`

`GET /chatbot/conversations/:conversationId/messages`

Evidence: `crestpetsystem/client/src/app/consumer/dashboard/marcia/components/consumer-marcia-chat.tsx:20-25`

#### What MCP chat currently supports

Rehydrates state via [http://Socket.IO](http://Socket.IO) :

`conversation_snapshot` includes messages/status/toolExecutions (`marcia-omnichannel-mcp/src/websocket.ts:475-518`)

Ignores attachments:

`files?: unknown[]; // Ignoring file uploads for now` (`marcia-omnichannel-mcp/src/websocket.ts:22-27`)

#### Minimum viable integration (two acceptable patterns)

**Pattern 1 (fastest UI change)**:

Keep conversation creation/list in Crest (so the UI can keep its sidebar).

Update Crest UI to source messages from MCP:

Subscribe to `conversation_snapshot`

Stop calling `GET /chatbot/conversations/:id/messages` when the socket target is MCP

Temporarily disable file upload UI when socket target is MCP (until MCP supports attachments).

**Pattern 2 (best UX, more work)**:

Make MCP the source of truth for both conversation list + messages:

Add MCP HTTP endpoints for listing user conversations (backed by `ConversationIndexService.getUserConversations` in `marcia-omnichannel-mcp/src/identity/conversation-index-service.ts:41-50,210-228`)

Provide an OpenAPI/Orval path to consume them (either by adding a second Orval target or proxying via Crest backend).

#### Message shape mismatch (implementer heads-up)

Crest UI message DTO is `MessageResponseDto` (from Crest OpenAPI), while MCP emits `{ role, content, createdAt }` in `conversation_snapshot`.

If you take Pattern 1, implement a client mapping layer:

`role === "user"` → `isUser: true`

`createdAt` → `timestamp`

attachments: **unsupported** until MCP implements file ingestion

## Key risks and mitigations

**R1 ,  API contract scope creep**: 29 tool calls → large endpoint surface.

**Mitigation**: Phase the internal API (IB-3 Phase 1 minimum set first), feature-flag tool availability in MCP prompts/tools.

**R2 ,  Tenant isolation**: Internal endpoints that accept `userId` risk cross-tenant access.

**Mitigation**: Enforce tenant isolation either via explicit tenant identifiers or via `TenantUser` joins (IB-3). Add audit logs for `userId`, tenant identifiers (when present), and scopes.

**R3 ,  Voice signature validation false negatives** (JSON stringify vs raw body).

**Mitigation**: Capture raw request body for validation (or confirm ElevenLabs signing canonicalization); test in prestaging with signature required.

**R4 ,  Rollback complexity** (client + server cutover).

**Mitigation**: Single env flag to route chat socket to Crest backend; keep Crest gateway intact until MCP is stable.

**R5 ,  Secrets drift** (Terraform ignores secret_string and container_definitions changes).

**Mitigation**: Document manual secret update + “force new deployment” step; add runbook checklist.
