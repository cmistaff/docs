<Note>
This is the canonical environment-variable guide for CrestPetSystem local development. The `.env.example` files in the repo are the source of truth for which variables exist; this page tells you how to use them and where to get values.
</Note>

## Source of Truth

`.env.example` is the **single source of truth** for environment variables. Per the April 2026 audit (CPE-5678):

- 40+ deprecated variables were removed from `.env.example`.
- 22 missing-but-used server variables were added.
- Variables are grouped semantically with section headers (App, Database, JWT, Redis, Twilio, AWS, AI, etc.) so it's easy to find what you need.

Per workspace, two files matter:

| File | Scope |
|------|-------|
| `client/.env.example` | Next.js frontend variables (includes `NEXT_PUBLIC_*` browser-exposed values) |
| `server/.env.example` | NestJS backend variables (includes secrets, DB URLs, third-party API keys) |

**Treat `.env.example` as read-only.** When you add a new variable in code, update the relevant `.env.example` in the same PR. Do not commit `.env`.

## Generating Your Local `.env`

### Step 1: Copy the example files

```bash
cd ~/path/to/crestpetsystem
cp client/.env.example client/.env
cp server/.env.example server/.env
```

### Step 2: Populate values

You have two options. Prefer Option A because it gives you working dev credentials immediately and stays in sync with the rest of the team.

#### Option A: Pull from AWS Secrets Manager (recommended)

Centralized local-dev secrets live in AWS Secrets Manager. INF-156 (April 9, 2026) moved every shared API key, token, and third-party credential into two secrets, so a single SSO-authenticated pull populates both `.env` files.

Once your AWS SSO is configured (see [AWS SSO Setup](/onboarding/aws-sso-setup)):

```bash
# Login (once per session)
aws sso login --sso-session crest

# Fetch the API/server secrets bundle
AWS_PROFILE=crest-prestaging-app-tester \
  aws secretsmanager get-secret-value \
  --secret-id local/crest-pet/api \
  --region us-east-1 \
  --query SecretString --output text \
  | jq -r 'to_entries[] | "\(.key)=\(.value)"' > server/.env

# Fetch the web/client secrets bundle
AWS_PROFILE=crest-prestaging-app-tester \
  aws secretsmanager get-secret-value \
  --secret-id local/crest-pet/web \
  --region us-east-1 \
  --query SecretString --output text \
  | jq -r 'to_entries[] | "\(.key)=\(.value)"' > client/.env
```

This overwrites your local `.env` files with the team's shared dev values for every populated key.

<Warning>
The two secrets only carry shared **third-party** credentials (API keys, webhook secrets, tokens). Local-only values that depend on your own dev box (`/etc/hosts` portal mappings, `APP_URL`, `PRIMARY_TENANT_ID` from your seeded DB, etc.) still come from `.env.example` defaults or your local setup. Re-apply those after the pull, or merge selectively rather than overwriting.
</Warning>

#### Option B: Ask in `#crestpet-onboarding`

If you cannot pull from Secrets Manager yet (SSO not provisioned, away from your work network, etc.), drop a message in `#crestpet-onboarding` and a teammate will share working dev values for any specific variables you're stuck on. Do not paste secrets back into the channel; use a DM or 1Password for the response.

### Step 3: Validate

Most server-side variables are validated at boot via the NestJS config schema. If a required variable is missing or malformed, the server fails fast with a descriptive error. The client validates `NEXT_PUBLIC_*` at build time. Read the error message before changing values blindly.

## Local-Only Values You Set Yourself

These are not in Secrets Manager because they depend on your own dev box and seeded database. Use the defaults from `.env.example` unless you know better.

| Variable | Typical local value |
|----------|---------------------|
| `DATABASE_URL` | `postgresql://app_user:app_password@localhost:5432/postgres` |
| `DIRECT_URL` | `postgresql://postgres:postgres@localhost:5432/postgres` |
| `REDIS_URL` | `redis://localhost:6379` |
| `APP_URL` | Your dev box URL or an ngrok tunnel - used for webhook callbacks |
| `PRIMARY_TENANT_ID` | The seeded clinic UUID (default seed: Pawsome Veterinary Clinic) |
| `NEXT_PUBLIC_API_HOST_MAP` | JSON map of portal subdomains to API subdomains |
| `APP_API_HOST_MAP` | Server-side equivalent of `NEXT_PUBLIC_API_HOST_MAP` |

Portal subdomain routing is finicky. If login silently fails or cookies don't set, the cause is almost always a mismatched API host map plus missing `/etc/hosts` entries. Ask in `#crestpet-onboarding` for the standard local map, or check the engineering handbook.

## Production and Staging Values

You should not need production or staging credentials on your dev box. ECS reads them from environment-specific Secrets Manager paths (`prestaging/crest-pet/api`, `staging/crest-pet/api`, etc.), not from `local/crest-pet/*`. If you genuinely need a non-local value for debugging, ask a team lead - most workflows can be debugged against prestaging.

## Adding a New Environment Variable

1. Add the variable name (with an empty value or a safe default) to the relevant `.env.example`.
2. Reference it through the appropriate config service (`CommonConfigService` server-side, `client/src/common/config/index.ts` client-side). Do not call `process.env` directly in feature code.
3. If the value is a secret, ask a team lead with `SecretsManager-Prestaging` to add it to `local/crest-pet/api` or `local/crest-pet/web`:
   ```bash
   AWS_PROFILE=crest-prestaging-secrets-manager \
     aws secretsmanager put-secret-value \
     --secret-id local/crest-pet/api \
     --region us-east-1 \
     --secret-string '{"NEW_KEY":"value", ...}'
   ```
4. Mention the new variable in `#crestpet-onboarding` so teammates know to re-pull.
