<Note>

By the end of this page you'll be ready to [run CrestPetSystem locally](/onboarding/run-crestpetsystem-locally).

You'll do these six steps in order. Each takes 2–5 minutes.

1. AWS SSO
2. GitHub authentication
3. Node version
4. Clone repos
5. Pull shared dev secrets
6. Verify your shell environment

## 1. AWS SSO {#aws-sso}

Your dev box needs AWS credentials so the crestpetsystem repo's `pnpm env:sync` script can pull shared dev secrets from AWS Secrets Manager, and so you can read prestaging logs, ECS state, and CloudWatch.

<Warning>
Region is **`us-east-1`**, not `us-east-2`. The dev EC2 boxes themselves live in `us-east-2` (Ohio), but the application accounts, KMS keys, and Secrets Manager entries you'll touch are in `us-east-1`. This trips up almost every new hire.
</Warning>

<Note>
**SSO group prerequisite.** Pulling secrets requires membership in the `AppTester-Prestaging` Identity Center group. If `pnpm env:sync` later returns `AccessDenied`, you are not in that group yet — ping `#crestpet-onboarding` and a teammate will get you added. The full group catalog is at [AWS SSO Setup → Step 1](/onboarding/aws-sso-setup#step-1-get-added-to-the-sso-groups).
</Note>

Run the SSO configurator:

```bash
aws configure sso
```

You'll be prompted for:

- **SSO session name:** `crest`
- **SSO start URL:** `https://cammarata.awsapps.com/start`
- **SSO region:** `us-east-1`
- **SSO registration scopes:** accept the default (`sso:account:access`)
- **CLI default client Region:** `us-east-1`
- **CLI default output format:** `json`
- **Profile name:** `crest-prestaging-app-tester`

Then log in. **Your dev box is headless — there is no browser to open — so use the device-code flow:**

```bash
aws sso login --profile crest-prestaging-app-tester --use-device-code
```

The terminal prints a URL and a short code. Open the URL on your laptop browser, paste the code, approve the device. The dev-box terminal continues automatically. Confirm it worked:

```bash
aws sts get-caller-identity --profile crest-prestaging-app-tester
```

You should see your SSO-assumed role ARN ending in `AppTester-Prestaging/<your-name>@creststaff.com`. If you get `Unable to locate credentials`, re-run the `aws sso login --use-device-code` command above.

For the deeper walkthrough (multi-environment profiles, group memberships, troubleshooting), see [AWS SSO Setup](/onboarding/aws-sso-setup).

## 2. GitHub authentication {#github}

The `gh` CLI is preinstalled on your dev box. Authenticate it:

```bash
gh auth login
```

When prompted:

- **What account?** GitHub.com
- **Preferred protocol?** HTTPS
- **Authenticate Git with your GitHub credentials?** Yes
- **How would you like to authenticate?** Login with a web browser (or paste a PAT if you've already created one)

Confirm:

```bash
gh auth status
gh repo list cmistaff --limit 5
```

Org access to `cmistaff` is granted at hire. If `gh repo list cmistaff` returns nothing, ping `#crestpet-onboarding` — your invite likely needs accepting.

## 3. Node version {#node}

Node 22 is the team baseline. Your shell starts up with NVM hooked in, so:

```bash
nvm use 22
node --version
```

You should see `v22.x.x`. If `nvm` isn't found, open a new shell — the global hook only attaches to fresh sessions.

## 4. Clone repos {#clone-repos}

All Crest repos live under `~/Crest/GitHub/` on your dev box, grouped by their parent system. Start with the main product (you'll need it for Step 5):

```bash
mkdir -p ~/Crest/GitHub/crestsystem
cd ~/Crest/GitHub/crestsystem
gh repo clone cmistaff/crestpetsystem
```

Other repos you may need depending on your role:

| Repo | Path | When you need it |
|------|------|------------------|
| `cmistaff/crestpetsystem` | `~/Crest/GitHub/crestsystem/crestpetsystem` | Main product. Everyone clones this. |
| `cmistaff/infra` | `~/Crest/GitHub/crestsystem/infra` | Backend/DevOps. Production Terraform. |
| `CrestLabsHQ/EC2DevsCrestPet` | `~/Crest/GitHub/EC2DevsCrestPet` | DevOps only. Manages the dev workstations. |
| `CrestCorp/AppCrestPetConsumer` | `~/Crest/GitHub/AppCrestPetConsumer` | Mobile/Marcia work. Expo + Express. |

Each repo has its own `CLAUDE.md` and `README.md` that explain local setup specifics.

## 5. Pull shared dev secrets {#secrets}

Shared local-dev credentials (API keys, tokens, webhook secrets for every third-party we use) live in AWS Secrets Manager and are pulled by a one-line `pnpm` script that reads your SSO session and writes the values into `.env` files inside the repo.

<Note>
**Contractor dev boxes do not have `~/management-tools/`.** That toolkit is internal to the operations machine. The only step you need is the `pnpm` command below.
</Note>

From inside the `crestpetsystem` repo:

```bash
cd ~/Crest/GitHub/crestsystem/crestpetsystem
AWS_PROFILE=crest-prestaging-app-tester pnpm env:sync:force --copy-to-env
```

What this does:

- Reads `local/crest-pet/api` and `local/crest-pet/web` from AWS Secrets Manager (us-east-1).
- Writes the values into `server/.env.local` + `client/.env.local`, and (because of `--copy-to-env`) also into `server/.env` + `client/.env` — which the app actually reads.
- Backs up any pre-existing `.env` to `.env.bak` first, so the command is **safe to re-run** when secrets rotate or new keys are added.
- Preserves any personal overlays you've set in `.env.user` (e.g. your NGROK token, personal API keys) — those always win over Secrets Manager values.

If you see:

- `AccessDenied` — your SSO user is not yet in the `AppTester-Prestaging` group. Ping `#crestpet-onboarding`.
- `expired token` / `SSO session has expired` — re-run `aws sso login --profile crest-prestaging-app-tester --use-device-code`, then re-run `pnpm env:sync:force --copy-to-env`.

For the full env-sync mechanism and the catalog of variables, see [Environment Variables](/onboarding/environment-variables).

## 6. Verify your shell environment {#verify}

Quick checklist. Run these in order. Every line should produce output as described.

```bash
aws sts get-caller-identity --profile crest-prestaging-app-tester
gh auth status
node --version
grep -c '^STRIPE_SECRET_KEY=sk_' ~/Crest/GitHub/crestsystem/crestpetsystem/server/.env
```

Expected results:

- `aws sts get-caller-identity` returns your SSO-assumed role ARN.
- `gh auth status` shows logged in to github.com as your username.
- `node --version` shows `v22.x.x`.
- The final `grep -c` returns `1` — confirming `server/.env` was populated with a `STRIPE_SECRET_KEY` value. If it returns `0`, re-run `pnpm env:sync:force --copy-to-env` from inside the `crestpetsystem` repo.

If all four pass, you're done. Next up: [Run CrestPetSystem Locally](/onboarding/run-crestpetsystem-locally).
