## Overview

The Crest Pet System uses a **three-tier branch strategy** with automated CI/CD pipelines built on GitHub Actions. The system ensures code quality through automated testing, linting, and security checks while maintaining a clean, linear git history.

### Key Principles

| Principle | Description |
| --- | --- |
| 🔀 Fast-Forward Only | All promotions use --ff-only merges to maintain linear history |
| 🚀 Automated Deployments | Push to any environment branch triggers deployment automatically |
| 🔒 Environment Isolation | Strict secret isolation between environments |
| 🔥 Hotfix Support | Emergency fixes can bypass normal flow with automatic sync-back |
| ✅ Quality Gates | Linting, testing, and validation before every promotion |
| 📦 Squash Merges | All PRs should use squash merge for clean, atomic commits |

### Environment URLs

| Environment | API URLs | Web URLs | Branch |
| --- | --- | --- | --- |
| 🔵 PreStaging | api.prestaging.vet, api.prestaging.pet, api.prestaging.express | prestaging.vet, prestaging.pet, prestaging.express | prestaging |
| 🟡 Staging | api.creststaging.vet, api.creststaging.pet, api.creststaging.express | creststaging.vet, creststaging.pet, creststaging.express | staging |
| 🟢 Production | api.crest.vet, api.crest.pet, api.crest.express | crest.vet, crest.pet, crest.express | main |

## Branch Strategy

### Branch Hierarchy

| Level | Branch | Environment | Promotion Method |
| --- | --- | --- | --- |
| 🟢 Top | main | Production | ⬆️ From staging (manual + approval) |
| 🟡 Middle | staging | Staging | ⬆️ From prestaging (manual) |
| 🔵 Base | prestaging | PreStaging | ⬆️ From feature branches (PR) |
| 🌿 Feature | PW-XXX-* | Local | Created from prestaging |

**Flow Direction:**

```
🌿 Feature Branch → 🔵 prestaging → 🟡 staging → 🟢 main
 ↓ ↓ ↓ ↓
 Local PreStaging Staging Production

```

### Branch Rules

| Branch | Purpose | Direct Push | PR Target | Protection |
| --- | --- | --- | --- | --- |
| main | Production code | ❌ Never | Only via staging promotion or ProdHotFix | 🔒 Highest |
| staging | Pre-production testing | ❌ Never | Only via prestaging promotion or StagingHotFix | 🔒 High |
| prestaging | Development integration | ❌ Never | All feature branches | 🔒 Standard |
| PW-XXX-* | Feature development | ✅ Yes | prestaging only | 🔓 None |

## Merge Rules

### PR Target Rules

| PR Source | PR Target | Allowed? | Condition |
| --- | --- | --- | --- |
| Feature branch | prestaging | ✅ Yes | Always (unless freeze active) |
| prestaging | staging | ✅ Yes | Promotion PRs only |
| staging | main | ✅ Yes | Promotion PRs only |
| Feature branch | staging | ⚠️ Conditional | Only with StagingHotFix: prefix |
| Feature branch | main | ⚠️ Conditional | Only with ProdHotFix: prefix |
| Feature branch | staging | ❌ No | Auto-retargeted to prestaging |
| Feature branch | main | ❌ No | Auto-retargeted to prestaging |

### PR Title Prefixes (Special)

| Prefix | Purpose | Target Branch | Bypasses Freeze |
| --- | --- | --- | --- |
| StagingHotFix: | Urgent staging fix | staging | ✅ Yes |
| ProdHotFix: | Critical production fix | main | ✅ Yes |
| StagingSyncBack: | Manual sync-back PR | staging | ✅ Yes |

### Required Merge Types

> ⚠️ **IMPORTANT**: Use the correct merge type based on PR type to maintain clean git history.

| PR Type | Merge Method | Reason |
| --- | --- | --- |
| Feature PR → prestaging | 📦 Squash and Merge | Creates single atomic commit, clean history |
| Hotfix PR → staging/main | 📦 Squash and Merge | Single commit for easy cherry-pick during sync |
| Promotion PR | 🔀 Fast-Forward (workflow) | Use promotion workflow, not PR merge |
| Sync-back PR (StagingSyncBack:) | 🔀 Merge Commit | Preserves commit SHAs, ties histories together |

**Why Squash Merge for Features/Hotfixes?**

| Benefit | Description |
| --- | --- |
| 📜 Clean History | Multiple WIP commits become one meaningful commit |
| ⚛️ Atomic Changes | Each feature/fix is a single commit |
| ↩️ Easy Rollback | Revert one commit instead of many |
| 🔄 Hotfix Sync | Single commits are easier to cherry-pick automatically |

**Why Merge Commit for Sync-back?**

| Benefit | Description |
| --- | --- |
| 🔗 Preserves SHAs | Original commit hashes remain intact |
| 🌳 Ties Histories | Creates a merge point showing where branches aligned |
| 🛡️ Prevents Divergence | Proper merge prevents further git history issues |

**Merge Button Selection:**

| Option | When to Use |
| --- | --- |
| ✅ Squash and merge | Feature PRs, Hotfix PRs |
| ✅ Create a merge commit | Sync-back PRs only |
| ❌ Rebase and merge | Avoid (rewrites history) |

### Automatic PR Retargeting

PRs targeting `main` or `staging` are automatically retargeted to `prestaging` unless:

- PR is a valid promotion (`prestaging` → `staging` or `staging` → `main`)
- PR title starts with a hotfix prefix (`StagingHotFix:` or `ProdHotFix:`)
- PR title starts with sync-back prefix (`StagingSyncBack:`)


When retargeted, the PR receives:

- 💬 An explanatory comment explaining the change
- 🏷️ A `retargeted-to-prestaging` label

## Deployment Process

### Automatic Deployments

| Trigger | Deploys To | Environment |
| --- | --- | --- |
| Push to prestaging | 🔵 PreStaging | api.prestaging.* |
| Push to staging | 🟡 Staging | api.creststaging.* |
| Push to main | 🟢 Production | api.crest.* |

### Smart Change Detection

Workflows use intelligent change detection to determine if deployment is needed:

**Web Application Triggers:**

- ✅ Changes in `client/**`
- ✅ Changes in `packages/common/**`
- ✅ Changes to `client/Dockerfile`, `client/next.config.*`, etc.


**API Application Triggers:**

- ✅ Changes in `server/**`
- ✅ Changes in `packages/common/**`
- ✅ Changes to `server/Dockerfile`, `server/nest-cli.json`, etc.

### Deployment Pipeline

| Stage | Action | Details |
| --- | --- | --- |
| 1️⃣ Detect | Analyze changes | Check git diff, determine if deploy needed |
| 2️⃣ Build | Docker build | Build with caching, tag as {sha}-{env} |
| 3️⃣ Push | ECR upload | Push image to Amazon ECR |
| 4️⃣ Secrets | Fetch config | Get secrets from AWS Secrets Manager |
| 5️⃣ Deploy | ECS update | Update task definition, deploy to Fargate |
| 6️⃣ Wait | Stabilize | Wait for service health checks |
| 7️⃣ Notify | Slack | Send success/failure notification |

#### Staging → Main (Production) Promotion

| --- | --- |

**What happens:**

| Check | Description |
| --- | --- |

### ⚠️ Post-Promotion: Create GitHub Release

> After production promotion, you must **manually create a GitHub Release** from the generated tag.

#### Creating a GitHub Release (Post-Promotion)

| Step | Action |
| --- | --- |
| 1️⃣ | Go to Repository → Releases → Draft a new release |
| 2️⃣ | Click "Choose a tag" and select the new tag (e.g., rel-2025.01.15-14.30Z) |
| 3️⃣ | Set Release title: Same as promotion title or more descriptive |
| 4️⃣ | Write Release notes: Summary, JIRA tickets, breaking changes, migrations |
| 5️⃣ | Check "Set as latest release" |
| 6️⃣ | Click "Publish release" |

**Via CLI:**

```
# List recent tags to find the new one
git tag -l 'rel-*' --sort=-creatordate | head -5

# Create release from tag
gh release create rel-2025.01.15-14.30Z \
 --title "Release: Assessment Filtering Fixes" \
 --notes "## Changes
- [PW-644] Fix assessment filtering
- [PW-645] Improve search performance

## JIRA Tickets
- PW-644, PW-645"

```

> 💡 **Why manual?** GitHub Releases allow for curated release notes, attached assets, and proper changelog formatting that benefits from human review.

## Hotfix Flow

### When to Use Hotfixes

| Scenario | Use Hotfix? |
| --- | --- |
| 🔴 Critical production bugs affecting users | ✅ Yes |
| 🔴 Security vulnerabilities | ✅ Yes |
| 🔴 Breaking functionality that can't wait | ✅ Yes |
| 🟡 Minor bugs or improvements | ❌ No - use normal flow |
| 🟡 New features | ❌ No - use normal flow |

### Staging Hotfix Flow

| Step | Action | Result |
| --- | --- | --- |
| 1️⃣ | Create branch from staging | git checkout staging && git checkout -b hotfix-critical-fix |
| 2️⃣ | Make fix and commit | git commit -m "fix: resolve critical bug" |
| 3️⃣ | Create PR with prefix | gh pr create --base staging --title "StagingHotFix: Fix critical bug" |
| 4️⃣ | Merge PR | 🚀 Auto-deploys to Staging |
| 5️⃣ | Automatic sync | hotfix-sync.yml triggers |

**Automatic Sync Process:**

| Step | Action |
| --- | --- |
| 🔍 | Detect StagingHotFix: prefix |
| 🔒 | Freeze prestaging |
| 🔄 | Sync prestaging from staging |
| 🔓 | Unfreeze prestaging |
| ✅ | Git history aligned! |

**Commands:**

```
# 1. Create hotfix branch from staging
git checkout staging && git pull origin staging
git checkout -b hotfix-critical-fix

# 2. Make fix and commit
git add .
git commit -m "fix(payment): resolve critical payment bug"

# 3. Create PR with StagingHotFix prefix
git push origin hotfix-critical-fix
gh pr create --base staging --title "StagingHotFix: Fix critical payment bug"

```

### Production Hotfix Flow

| Step | Action | Result |
| --- | --- | --- |
| 1️⃣ | Create branch from main | git checkout main && git checkout -b hotfix-security-fix |
| 2️⃣ | Make fix and commit | git commit -m "fix: patch security vulnerability" |
| 3️⃣ | Create PR with prefix | gh pr create --base main --title "ProdHotFix: Fix security issue" |
| 4️⃣ | Merge PR | 🚀 Auto-deploys to Production |
| 5️⃣ | Automatic sync | hotfix-sync.yml triggers |

**Automatic Sync Process:**

| Step | Action |
| --- | --- |
| 🔍 | Detect ProdHotFix: prefix |
| 🔒 | Freeze staging and prestaging |
| 🔄 | Sync staging from main |
| 🔄 | Sync prestaging from staging |
| 🔓 | Unfreeze all branches |
| ✅ | All environments synchronized! |

**Commands:**

```
# 1. Create hotfix branch from main
git checkout main && git pull origin main
git checkout -b hotfix-security-fix

# 2. Make fix and commit
git add .
git commit -m "fix(auth): patch security vulnerability"

# 3. Create PR with ProdHotFix prefix
git push origin hotfix-security-fix
gh pr create --base main --title "ProdHotFix: Fix security vulnerability"

```

### Hotfix Sync Algorithm

| Condition | Sync Method |
| --- | --- |
| Target is ancestor of source | 🔀 Fast-forward merge |
| Branches have diverged | 🍒 Cherry-pick hotfix commits only |

### Manual Sync-Back (Fallback)

If automatic sync fails, create a manual sync-back PR:

```
# Create sync-back PR
gh pr create --base staging --head prestaging \
 --title "StagingSyncBack: Sync hotfix commits from prestaging"

```

> ⚠️ Remember: Sync-back PRs must use **Merge Commit**, not Squash!

## Protection Mechanisms

### Overview

| Protection | Purpose | Method |
| --- | --- | --- |
| 🔄 Concurrency Control | Prevents simultaneous promotions | Workflow queue |
| 🔒 Freeze Mechanism | Blocks merges during hotfix sync | Repository variables |
| 🛡️ Release Branch Guard | Validates PRs to staging/main | Required check |
| 🔀 Auto-Retargeting | Redirects PRs to correct branch | Automatic workflow |

### 1. Concurrency Control

Prevents concurrent workflow runs for promotions:

```
concurrency:
 group: promote-to-staging # or promote-to-main
 cancel-in-progress: false # Queue instead of cancel

```

**Protects:** Multiple users running promotion workflows simultaneously

### 2. Freeze Mechanism

Temporarily blocks PR merges during hotfix synchronization:

| Variable | Set By | Checked By | Purpose |
| --- | --- | --- | --- |
| FREEZE_PRESTAGING | hotfix-sync.yml | release-branch-guard.yml | Block prestaging merges |
| FREEZE_STAGING | hotfix-sync.yml | release-branch-guard.yml | Block staging merges |

**Bypass:** Add `override-freeze` label to PR

### 3. Release Branch Guard

Validates all PRs targeting `staging` or `main`:

| Scenario | Result |
| --- | --- |
| Valid promotion (prestaging → staging) | ✅ Allowed |
| Valid promotion (staging → main) | ✅ Allowed |
| Hotfix PR with proper prefix | ✅ Allowed |
| Random feature branch | ❌ Blocked |
| PR during freeze (no override label) | ⚠️ Blocked |

### 4. Auto-Retargeting

Automatically redirects PRs to correct branch:

| Original Target | New Target | Action |
| --- | --- | --- |
| main | prestaging | Retargeted + comment + label |
| staging | prestaging | Retargeted + comment + label |

### Protection Summary

| Scenario | Protection | Method |
| --- | --- | --- |
| Concurrent promotions | ✅ Prevented | Concurrency control |
| Hotfix sync conflicts | ✅ Prevented | Freeze mechanism |
| Wrong PR target | ✅ Auto-corrected | Retargeting workflow |
| Invalid PR to release branch | ✅ Blocked | Release branch guard |

## Rollback Procedures

### Option 1: Rollback to Previous Commit (Fast)

```
# Get previous commit SHA from promotion summary
PREVIOUS_SHA="abc123..."

# Rollback
git checkout main # or staging
git reset --hard $PREVIOUS_SHA
git push origin main --force-with-lease

```

### Option 2: Rollback to Release Tag

```
# List recent release tags
git tag -l 'rel-*' --sort=-creatordate | head -5

# Rollback to specific tag
git checkout main
git reset --hard rel-2025.01.15-14.30Z
git push origin main --force-with-lease

```

### Rollback Decision Guide

| Situation | Action |
| --- | --- |
| Know the bad commit? | git reset --hard PREVIOUS_SHA |
| Don't know bad commit? | Find last good rel-* tag |
| After reset | git push --force-with-lease |
| After push | Verify deployment, notify team |

## Troubleshooting

### Common Issues

#### 1. Deployment Skipped Unexpectedly

**Symptoms:** "No relevant changes detected"

| Solution | Command/Action |
| --- | --- |
| Check trigger patterns | Verify changed files match patterns |
| Force deploy | Manual deployment with "Force deploy" ✓ |
| Check git diff | git diff HEAD~1 --name-only |

#### 2. PR Blocked by Freeze

**Symptoms:** "Staging/Prestaging is temporarily frozen"

| Solution | Command/Action |
| --- | --- |
| Wait | Hotfix sync may still be running |
| Override | Add override-freeze label to PR |
| Check status | View hotfix-sync.yml workflow |

#### 3. Promotion Fails - Branches Diverged

**Symptoms:** "Fast-forward not possible"

| Solution | Command/Action |
| --- | --- |
| Check force-push | Did someone force-push to target? |
| Verify sync | Was hotfix properly synced back? |
| Manual fix | May require manual resolution |

#### 4. Build Failures

**Symptoms:** Docker build fails

| Solution | Command/Action |
| --- | --- |
| Check logs | Review build logs for specific errors |
| Verify secrets | Ensure all required secrets are set |
| Update deps | Check if dependencies are current |
| AWS creds | Verify AWS credentials are valid |

#### 5. ECS Deployment Fails

**Symptoms:** Service doesn't stabilize

| Solution | Command/Action |
| --- | --- |
| ECS events | Check ECS service events in AWS Console |
| Task def | Verify task definition is valid |
| Logs | Check container logs in CloudWatch |
| Health check | Verify health check configuration |

#### 6. Slack Notifications Not Working

**Symptoms:** No Slack messages after deployment

| Solution | Command/Action |
| --- | --- |
| Secret | Verify SLACK_WEBHOOK_URL is set |
| URL valid | Check webhook URL is still valid |
| Logs | Review workflow logs for errors |
