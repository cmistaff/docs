## 🎯 Overview

This guide covers everything a new developer needs to access PostgreSQL and Redis data stores in our AWS infrastructure. We use **AWS IAM Identity Center (SSO)** for authentication and **AWS Systems Manager Session Manager** for secure bastion access - no SSH keys required!

## 🔧 Prerequisites & Installation

### Required Tools

| Tool | Purpose | Installation |
| --- | --- | --- |

## 🔐 AWS SSO Setup (One-Time)

### Step 1: Get Added to SSO Groups

Contact your team lead or DevOps to be added to the appropriate SSO groups:

| Access Level | SSO Groups | What You Get |
| --- | --- | --- |
| Developer | Observers-{Env} + AppDeployers-{Env} + DataManagers-{Env} | Full development access |
| Read-Only | Observers-{Env} + DataViewers-{Env} | View logs + read-only database |
| DevOps | InfraAdmins | Infrastructure management (no data access) |

**Environments:** `Prestaging`, `Staging`, `Production`

> ⚠️ **Production Access:** Most developers only get Observer access to production. Write access requires special approval.

### Step 2: Configure AWS CLI with SSO

Run the interactive setup:

```bash
aws configure sso
```

Enter the following when prompted:

- **SSO session name:** `crest`
- **SSO start URL:** `https://cammarata.awsapps.com/start`
- **SSO region:** `us-east-1`
- **SSO registration scopes:** `sso:account:access`


Then complete the browser login flow.

### Step 4: Test Your Access

```bash
# Login to SSO (opens browser)
aws sso login --sso-session crest

# Test each profile you have access to
aws sts get-caller-identity --profile crest-staging-observer
aws sts get-caller-identity --profile crest-staging-data
aws sts get-caller-identity --profile crest-staging-deployer
```

## 🚀 Connecting to Databases

### Clone the Infrastructure Repository

```bash
# Clone the infra repo (contains connection scripts)
git clone git@github.com:cmistaff/infra.git
cd infra

# Install dependencies
pnpm install
```

### Daily Workflow

```bash
# 1. Login to AWS SSO (once per day)
aws sso login --sso-session crest

# 2. Connect to PostgreSQL
pnpm bastion:postgres:staging:sso -p crest-staging-data # Staging
pnpm bastion:postgres:prestaging:sso -p crest-prestaging-data # Prestaging

# 3. Connect to Redis
pnpm bastion:redis:staging:sso -p crest-staging-data # Staging
pnpm bastion:redis:prestaging:sso -p crest-prestaging-data # Prestaging
```

### Connection Ports

| Service | Prestaging | Staging | Production |
| --- | --- | --- | --- |
| PostgreSQL | localhost:35432 | localhost:15432 | localhost:25432 |
| Redis | localhost:36379 | localhost:16379 | localhost:26379 |

### Available Commands (Quick Reference)

```bash
# PostgreSQL connections (with SSO)
pnpm bastion:postgres:prestaging:sso -p <profile>
pnpm bastion:postgres:staging:sso    -p <profile>
pnpm bastion:postgres:production:sso -p <profile>

# Redis connections (with SSO)
pnpm bastion:redis:prestaging:sso -p <profile>
pnpm bastion:redis:staging:sso    -p <profile>
pnpm bastion:redis:production:sso -p <profile>

# Read-only mode (viewer_iam user)
pnpm bastion:postgres:staging:reader -p <profile>

# Read-write mode (developer_iam user)
pnpm bastion:postgres:staging:writer -p <profile>
```

## 🛠️ Database Client Configuration

### DBeaver (PostgreSQL)

#### Step 1: Start the Tunnel

```bash
# Keep this terminal open while using DBeaver
pnpm bastion:postgres:staging:sso -p crest-staging-data
```

You'll see output like:

```text
✓ IAM AUTH: DEVELOPER (READ/WRITE + BYPASS RLS)

Connection:
  Host:     localhost
  Port:     15432
  Database: crest_pet_staging
  Username: developer_iam

IAM Token (valid 15 min): <TOKEN>
```

#### Step 2: Configure DBeaver Connection

1. Open DBeaver → **Database** → **New Database Connection**
2. Select **PostgreSQL**
3. Configure:


| Field | Value |
| --- | --- |
| Host | localhost |
| Port | 15432 (staging) or 35432 (prestaging) or 25432 (production) |
| Database | crest_pet_staging (or crest_pet_prestaging / crest_pet_production) |
| Username | developer_iam (from script output) |
| Password | Copy the IAM token from script output |

1. Click **Test Connection** → Should succeed!
2. Click **Finish**


> 💡 **Tip:** IAM tokens expire after 15 minutes, but established connections stay active. If disconnected, re-run the script to get a new token.

### Redis Insight

#### Step 1: Install Redis Insight

```bash
# macOS
brew install --cask redisinsight

# Or download from: https://redis.com/redis-enterprise/redis-insight/

# Also you can use Redis for VSCode extension
```

#### Step 2: Start the Tunnel

```bash
# Keep this terminal open while using Redis Insight
pnpm bastion:redis:staging:sso -p crest-staging-data
```

#### Step 3: Configure Redis Insight

1. Open Redis Insight → **Add Redis Database**
2. Configure:


| Field | Value |
| --- | --- |
| Host | localhost |
| Port | 16379 (staging) or 36379 (prestaging) or 26379 (production) |
| Name | Crest Pet Staging Redis |
| Username | Copy from bastion script output (e.g., crest-pet-staging-developer) |
| Password | Copy IAM token from bastion script output |
| Use TLS | ✅ Enabled (required) |

1. Click **Add Redis Database**


> 💡 **Redis Users:** The bastion script auto-detects your access level:
>
> - DataViewers get `crest-pet-{env}-viewer` (read-only)
> - DataManagers get `crest-pet-{env}-developer` (full access)

> 💡 **Note:** All connection credentials (username, password) are displayed in the bastion script output. No need to manually fetch secrets.

### Command Line Access

#### PostgreSQL (psql)

```bash
# Start tunnel in one terminal
pnpm bastion:postgres:staging:sso -p crest-staging-data

# In another terminal, connect with psql
PGPASSWORD='<TOKEN_FROM_SCRIPT>' psql -h localhost -p 15432 -U developer_iam crest_pet_staging

# Or using connection string
psql "postgresql://developer_iam:<TOKEN>@localhost:15432/crest_pet_staging"
```

#### Redis (redis-cli)

```bash
# Start tunnel in one terminal
pnpm bastion:redis:staging:sso -p crest-staging-data

# In another terminal, connect with redis-cli
# Use the username and IAM token from the bastion script output
redis-cli -h localhost -p 16379 --tls --insecure --user <USERNAME> --pass '<IAM_TOKEN>'

# Example for DataManager in staging:
redis-cli -h localhost -p 16379 --tls --insecure --user crest-pet-staging-developer --pass '<IAM_TOKEN>'

# Test connection
redis-cli -h localhost -p 16379 --tls --insecure --user crest-pet-staging-developer --pass '<IAM_TOKEN>' PING
```

> ⚠️ **TLS Required:** Always use `--tls --insecure` flags. `--insecure` is needed because the cert is for `*.cache.amazonaws.com`, not `localhost`.

## 🔍 Understanding Access Levels

### IAM Database Authentication

We use **IAM Database Authentication** for PostgreSQL - no shared passwords!

| SSO Group | PostgreSQL User | Access Level |
| --- | --- | --- |
| DataViewers-{Env} | viewer_iam | SELECT only, bypass RLS |
| DataManagers-{Env} | developer_iam | Full CRUD, bypass RLS |
| DataOwners-{Env} | developer_iam | Full access |

### Redis Authentication

Redis uses **IAM authentication** with environment-specific users:

| SSO Group | Redis User | Access Level |
| --- | --- | --- |
| DataViewers-{Env} | crest-pet-{env}-viewer | Read-only |
| DataManagers-{Env} | crest-pet-{env}-developer | Read/write |

> 💡 The bastion script auto-generates a 15-minute IAM token. No passwords needed!

### Token & Session Lifetimes

Understanding the different expiration times:

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        TOKEN/SESSION HIERARCHY                              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  SSO Session (2-12 hours)                                                   │
│  └── Your AWS CLI login session                                             │
│      └── Determines how long before you need to run `aws sso login` again   │
│                                                                             │
│  SSM Port Forwarding Session (until you close the terminal)                 │
│  └── The bastion tunnel (pnpm bastion:postgres:...)                         │
│      └── Stays open as long as the script runs                              │
│                                                                             │
│  IAM Auth Token (15 minutes)                                                │
│  └── The password/token used to authenticate to PostgreSQL/Redis            │
│      └── Only needed for INITIAL connection                                 │
│      └── Once connected, your session stays active!                         │
│                                                                             │
│  Database Connection (until you disconnect)                                 │
│  └── Your actual PostgreSQL/Redis connection                                │
│      └── Stays open even after IAM token expires                            │
│      └── Only closes when YOU close it or network drops                     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### Example Timeline

```text
0:00  - Run `aws sso login` → SSO session starts (4-12h depending on role)
0:01  - Run `pnpm bastion:postgres:staging:sso` → tunnel opens + 15-min token generated
0:02  - Connect DBeaver using the token → connection established ✅
0:16  - IAM token expires (but connection STAYS ACTIVE!)
2:00  - Still connected, running queries ✅
3:00  - Close DBeaver connection
3:01  - Need to reconnect? Run bastion script again for new token
4:00  - SSO session expires → need to run `aws sso login` again
```

#### SSO Session Durations (by Role)

Higher privilege = shorter session for security:

| Capability | Session Duration | Rationale |
| --- | --- | --- |
| Observer | 12 hours | Logs only, no data access |
| DataViewer | 8 hours | Read-only data, limited risk |
| AppDeployer | 8 hours | Deployment only, no data access |
| DataManager | 4 hours | Read/write data, higher risk |
| DataOwner | 2 hours | Full PII access, maximum protection |

> ⚠️ **Production has shorter sessions**: DataViewer is 4h (not 8h) and DataManager is 2h (not 4h) in production.

#### Key Takeaways

| What | Lifetime | What Happens When It Expires |
| --- | --- | --- |
| SSO Session | 2-12 hours | Run aws sso login --sso-session crest |
| Bastion Tunnel | Until you close it | Re-run the bastion script |
| IAM Token (PostgreSQL) | 15 minutes | Existing connections stay open; new need token |
| IAM Token (Redis) | 15 minutes | Existing connections stay open; new need token |
| DB Connection | Until disconnected | Just reconnect (get new token first) |

## 🐛 Troubleshooting

### Common Issues

#### "SSO session has expired"

```bash
# Re-login to SSO
aws sso login --sso-session crest
```

#### "No running bastion instance found"

The bastion host may be stopped. Contact DevOps to start it.

#### "Access Denied" when connecting

1. Verify you're in the correct SSO group
2. Check you're using the right profile for the environment
3. Try logging out and back in:
```bash
rm -rf ~/.aws/sso/cache
aws sso login --sso-session crest
```

#### "Port already in use"

```bash
# Find what's using the port
lsof -i :15432

# Kill the process
kill -9 <PID>

# Or use a different port
pnpm bastion:postgres:staging:sso -p crest-staging-data -l 15433
```

#### DBeaver "Connection refused"

1. Make sure the tunnel script is running (don't close the terminal!)
2. Verify you're connecting to `localhost`, not the RDS endpoint
3. Check the port matches the environment (15432 for staging)

#### "IAM token expired" or "Authentication failed"

IAM tokens are valid for **15 minutes**, but this only affects **new connections**:

- ✅ **Already connected?** Your session stays active even after token expires
- ❌ **Connection dropped?** You need a fresh token to reconnect


To reconnect:

1. Re-run the bastion script to get a new token
2. Copy the new IAM token from the script output
3. Reconnect DBeaver/Redis Insight with the new token


> 💡 **Pro tip**: If you get disconnected frequently, keep the bastion script terminal open. The tunnel stays active as long as the script runs.
