This channel is dedicated to end-of-day progress updates. Every developer must post an update before signing off. This keeps the entire team aligned on progress, blockers, and upcoming work.

1. Tasks you worked on today
2. Tasks completed today
3. Plans for tomorrow
4. **PR links** for all draft or ready-for-review work


Consistent daily updates help keep the entire team aligned. Also, don't forget to

**Daily Requirements**

- **Create a Draft PR** as soon as you start any ticket (targeting pre-staging).
- **Commit and push** your work before signing off every day, even if incomplete.

When you complete your assigned task, record a detailed Loom video of the feature or changes before moving your Draft PR to Ready for Review. The video should walk through all paths and edge cases. Share it in the channel and tag Michael. Once Michael reviews the Loom and has no change requests, you can move your Draft PR to Ready for Review.

### #crestpet-review

This channel manages our code review workflow.

- Once your loom is approved from Michael, Mark your PR as **Ready for Review**
- Resolve any AI-generated comments
- Share the PR in this channel with a short note
- Tag the reviewers for that day
- Every communication regarding that review should be done in the same thread

**Two approvals** are required before a PR can be merged into the **pre-staging** branch**For reviewers:**

- In crestpet review channel they have to give the 👀 icon when the reviewer has started looking at it and the ✅ icon when they have completed review.

### Connecting to Your Dev Box

1. **Install Tailscale** on your local machine: [https://tailscale.com/download](https://tailscale.com/download)
2. **Accept your Tailscale invite** - you'll receive an invite link via email or Slack. Click the link to join our network.
3. **Log in to Tailscale** - you can authenticate with:**Google** using your `@creststaff.com` or `@cmistaff.com` email
4. **GitHub** using your GitHub account
5. **Important: If you sign in via GitHub**, let IT know your GitHub username so they can update your access permissions. GitHub logins appear as `your-github-username@github` which needs to be added to your ACL.**If you see "cannot join another tenant"** when trying to sign in with Google, your email may already be associated with another Tailscale account. In this case, provide your **GitHub username** to **Michael Cammarata** so he can set up access via GitHub login instead.
6. **SSH to your dev box:**


```bash
ssh ubuntu@dev-yourname
```

Replace `yourname` with your assigned dev box name (e.g., `dev-michael`, `dev-affan`, `dev-serhii`).

### Using Cursor with Your Dev Box

2. Use the **Remote - SSH** extension

### Fetching Secrets for CrestPetSystem

Your dev box has IAM permissions to fetch secrets from AWS Secrets Manager:

```bash
# Get server environment variables
aws secretsmanager get-secret-value \
 --secret-id crest-dev/server-env \
 --region us-east-2 \
 --query SecretString --output text > server/.env

# Get client environment variables
aws secretsmanager get-secret-value \
 --secret-id crest-dev/client-env \
 --region us-east-2 \
 --query SecretString --output text > client/.env
```

### Troubleshooting

**Can't SSH to your dev box?**

1. Check Tailscale is running: `tailscale status`
2. Ping the instance: `tailscale ping dev-yourname`
3. If you get **"tailnet policy does not permit you to SSH to this node"**, your SSH ACL may be missing. Contact IT (Michael Cammarata) to add your identity to the SSH ACL.
4. If you signed in via GitHub, make sure IT has added your `github-username@github` identity to BOTH regular ACLs AND SSH ACLs.


**Instance not responding?**

- It may be hibernated. Contact IT to start it, or check AWS Console if you have access.

### For Admins: Adding a New Developer to Tailscale

When adding a new developer, you must configure **TWO separate ACL sections**:

1. `acls` section - Regular network/port access
2. `ssh` section - Tailscale SSH access (REQUIRED for `ssh ubuntu@dev-name`)


**Without both**, users get: `tailnet policy does not permit you to SSH to this node`

Always add ALL possible login identities for each user:

- `username@creststaff.com` (Google login)
- `username@cmistaff.com` (Google login)
- `github-username@github` (GitHub login)

### Best Practices

- **Commit and push your work regularly** - dev boxes can be rebuilt if needed
- **Don't store secrets in plain text files** - use the AWS Secrets Manager commands above
- **Use tmux for long-running processes** - keeps sessions alive if your SSH disconnects
- **Check disk space periodically** - run `df -h /` to monitor usage
- **Back up your configs with chezmoi** - so you can quickly restore after any rebuild

### nip.io (Local Subdomain Testing)

[nip.io](https://nip.io) is a wildcard DNS service for local development - it allows you to use subdomains that resolve to localhost without any configuration. **No account required**, it just works.

**How it works:** Any subdomain followed by an IP address resolves to that IP.

**Examples:**

- `clinic.127.0.0.1.nip.io` → resolves to `127.0.0.1`
- `consumer.127.0.0.1.nip.io` → resolves to `127.0.0.1`
- `anything.192.168.1.100.nip.io` → resolves to `192.168.1.100`


**When to use:** This is useful when testing multi-tenant features locally where different subdomains need to point to your local server, or when testing subdomain-based routing without modifying your hosts file.

## 7. Database Dump Usage (Local Development Without Seeding)

### Purpose

Until automated seeding scripts are finalized, developers must use a **pre-staging database dump** to initialize their local environment. This avoids manual data creation and ensures consistency with realistic data and system state.

### Source of Truth

The latest database dump is always available in the `#crestpet-onboarding` Slack channel:

- **Location:** Go to the channel's **Files** tab → look in the **"dumps"** folder
- **Source:** Pre-staging environment
- **File naming format:** `DD-MM-YY-dump.sql` (e.g., `18-12-25-dump.sql`)


**⚠️ Important:** New dumps are added periodically as the database evolves. **Always check for and use the latest dump file** to avoid issues with schema mismatches or authentication failures. If a newer dump has been uploaded since your last setup, re-import it to stay current.

### Local Setup Procedure

If seeding scripts are not present or not ready, follow the steps below:

1. Go to the `#crestpet-onboarding` Slack channel
2. Click on the **Files** tab and navigate to the **"dumps"** folder
3. Download the **latest database dump** (check the date in the filename)
4. Navigate to the **root directory** of the project
5. Run the following command to reset and import the database:


```
pnpm db:reset-and-import
```

**Example:**

```
pnpm db:reset-and-import ~/Downloads/18-12-25-dump.sql
```

This command will:

- Reset the local database
- Import the pre-staging data
- Allow development without recreating data manually

### Important Notes

- **Always use the latest dump** - older dumps may be incompatible with current authentication or schema changes (e.g., Better Auth migration)
- This approach is **temporary** and will be replaced once proper seeding scripts are available
- Always re-import a fresh dump after major schema or auth changes
- Check the `#crestpet-onboarding` channel periodically for announcements about new dumps

### Objective

Enable fast, consistent local development while maintaining alignment with real system data - without blocking progress on pending seeding automation.
