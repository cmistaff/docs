## Executive Summary {#executive-summary}

An audit of S3 file storage across the Crest Pet System revealed that **14 out of ~26 production code paths** that construct S3 object keys **do not include a **`tenantId`** prefix**. This means files from different tenants are stored in a flat or loosely organized namespace, creating three risks:

1. **No tenant-level isolation** - Without tenantId in the key path, S3 bucket policies cannot restrict access per tenant, and there is no structural boundary between tenants' files.
2. **No tenant-level lifecycle management** - Cannot apply S3 lifecycle rules, compliance holds, or data deletion per tenant (e.g., when a tenant is offboarded, there is no way to identify and purge their files via prefix).
3. **Audit and debugging difficulty** - When investigating a storage issue, there is no way to list all files belonging to a specific tenant without querying the database and cross-referencing S3 keys.


This document proposes a two-phase approach: (1) fix all new uploads to use tenantId-prefixed keys, and (2) migrate existing S3 objects to the new key format using data from the database.

## Problem Statement {#problem-statement}

### Why tenantId in S3 keys matters {#why-tenantid-in-s3-keys-matters}

Crest Pet System is a **multi-tenant SaaS platform**. The core security invariant is that every piece of tenant data must be explicitly scoped. For database queries, this is enforced via `WHERE tenantId = ?` in every repository method. However, this same discipline was not consistently applied to S3 object keys.

### What can go wrong without it {#what-can-go-wrong-without-it}

| Risk | Severity | Example |
| --- | --- | --- |
| Cannot purge a tenant's files on offboarding | High | Tenant cancels subscription; their files remain mixed with others, requiring DB queries to find them |
| Cannot apply S3 bucket policies per tenant | Medium | If we ever need IAM-based tenant isolation or S3 Access Points, tenantId-prefixed keys are a prerequisite |
| Key collisions in flat namespaces | Low | generateUniqueFileName() returns ${uuid}.ext - while UUIDs are unique, the flat structure makes it impossible to identify ownership without a DB lookup |
| Cannot use S3 Inventory / analytics per tenant | Medium | Billing, storage usage tracking, or compliance reporting per tenant requires prefix-based grouping |
| Cannot perform per-tenant cost analysis | High | Without tenantId-prefixed keys, there is no way to calculate storage costs per tenant. S3 Storage Lens and S3 Inventory can filter by prefix, enabling per-tenant storage size reports, request counts, and cost attribution - but only if keys are organized by tenant. This is critical for accurate margin analysis, identifying high-storage tenants, enforcing storage quotas, and building usage-based billing models. |

### What is NOT at risk {#what-is-not-at-risk}

- **Data leakage between tenants via S3 is NOT currently possible** - All access to S3 is mediated through signed URLs generated server-side, and the server always uses database records (which are tenant-scoped) to find the correct key. A tenant cannot guess another tenant's S3 key and access it directly.
- This is a **structural/operational improvement**, not an active security vulnerability.

## Target Architecture {#target-architecture}

### Key Format Convention {#key-format-convention}

```
# Single-feature bucket (bucket itself is the feature scope):
${tenantId}/${additional-context}/${uuid}.${ext}

# Multi-feature bucket (DOCUMENTS, MEDIA, CLINIC_DOCUMENTS):
${tenantId}/${feature-prefix}/${additional-context}/${uuid}.${ext}

```

### Why feature-prefix is only needed in shared buckets {#why-feature-prefix-is-only-needed-in-shared-buckets}

Seven of our 11 buckets serve a single feature (e.g., `TOOTH_SCANS` only holds tooth scan images). In these buckets, the bucket name already provides the feature context, so a feature prefix in the key is redundant. Only the 4 shared buckets (`DOCUMENTS`, `MEDIA`, `CLINIC_DOCUMENTS`, `VET_ASSISTANT_FILES`) need a feature prefix to distinguish different types of files.

## Special Cases - Cross-Tenant & Ambiguous Data {#special-cases-cross-tenant-ambiguous-data}

### Chatbot & SMS Files {#chatbot-sms-files}

**Models:** `ChatbotFiles`, `SmsFiles` **Problem:** Neither model has `tenantId`. The chain is `ChatbotFiles` → `ChatbotMessage` → `ChatbotConversations` → `userId` (no tenant). Same for SMS.

**Decision: No **`tenantId`** column. Accept user-scoped isolation.**

These features are **tenant-agnostic by design**. The Marcia AI agent is a consumer-facing conversational assistant accessible via WhatsApp, phone calls, and web chat - channels where no tenant context exists. A user can talk to Marcia without being associated with any specific clinic or tenant. The `userId` is the natural isolation boundary, not `tenantId`.

**Implications for S3 keys:**

- `AGENT_CONVERSATION_IMAGES` bucket remains user-scoped (not tenant-scoped)
- Current key patterns (`${uuid}.${ext}` for chatbot, `${userId}/${conversationId}/${filename}` for SMS) are acceptable
- No schema changes required for `ChatbotConversations` or `SmsConversation`
- No S3 key migration needed for these files
- The `AGENT_CONVERSATION_IMAGES` bucket is an **exception** to the `tenantId`-prefix rule, documented here as an accepted exclusion

### Networks {#networks}

**Model:** `Network` has no `tenantId`. Ownership is via `NetworkOwnerTenants` (many-to-many).

**Recommendation:** When a network is created, the creating tenant's ID is known. Use `NetworkOwnerTenants WHERE isMasterOwner = true` as the canonical owner for S3 key purposes. The logo is uploaded by the master owner, and displayed to all member clinics - but the file belongs to the creator.

**Migration query:**

```
SELECT n.id, n."networkLogoPath", not2."ownerTenantId"
FROM "Network" n
JOIN "NetworkOwnerTenants" not2 ON not2."networkId" = n.id AND not2."isMasterOwner" = true
WHERE n."networkLogoPath" IS NOT NULL;

```

### Referral Documents {#referral-documents}

**Model:** `PatientReferralDocuments` → `PatientReferral` has `fromClinicId` and `toClinicId` (two different tenants).

**Recommendation:** Scope to the **originating clinic** (`fromClinicId`). The referring clinic uploads the documents. Both clinics can access them via signed URLs (as today), but the S3 key is organized under the uploader's tenant.

**Migration query:**

```
SELECT prd.id, prd."documentPath", pr."fromClinicId" AS "tenantId"
FROM "PatientReferralDocuments" prd
JOIN "PatientReferral" pr ON prd."referralId" = pr.id
WHERE pr."fromClinicId" IS NOT NULL;

```

### Document Model {#document-model}

**Model:** `Document` has no `tenantId`. Has `petId` → `Pet` → `PetTenant`.

**Important:** A pet can belong to multiple tenants (via `PetTenant`). For consumer-uploaded documents, the uploading user's tenant context should be used.

**Migration query:**

```
SELECT d.id, d."documentPath", pt."tenantId"
FROM "Document" d
JOIN "PetTenant" pt ON d."petId" = pt."petId"
WHERE d."petId" IS NOT NULL
ORDER BY pt."createdAt" ASC
LIMIT 1; -- Use earliest tenant association

```

> **Edge case:** Documents with `petId = NULL` cannot be resolved. These should be logged and handled manually or left in place with a `_unscoped/` prefix.

### SendGrid Inbound Email Attachments {#sendgrid-inbound-email-attachments}

**Problem:** Email arrives at a `@Public()` webhook endpoint (`POST /webhooks/sendgrid/inbound`) with no tenant context. Attachments are currently uploaded to S3 *before* the tenant is resolved.

**Decision: Reorder - resolve tenant before upload (Option A).**

**Analysis of current flow:**

1. `SendGridInboundProcessorService.processInboundEmail()` parses the email and extracts attachments
2. Attachments are uploaded to S3 immediately (no `tenantId`)
3. S3 keys are passed to `InsuranceEmailReplyService.processEmailReply()`
4. `findMatchingRequest(emailFrom)` matches the sender email → `InsuranceProviderContactRequest` (which has `tenantId`)
5. If no match found, the method returns early - **attachments were uploaded for nothing**


**Why reordering works:**

- The tenant lookup (`findMatchingRequest`) is a synchronous indexed Prisma query on `insuranceProvider.email` - fast and reliable
- It runs in the same request lifecycle (not async/background)
- When no match is found, attachments are already discarded (early return) - uploading first wastes S3 writes
- `InsuranceProviderContactRequest` always has a `tenantId` column


**Required refactor:**

1. Move `findMatchingRequest` (or a lightweight tenant-resolving method) into `SendGridInboundProcessorService`, or call it before `uploadAttachmentsToS3`
2. If match found → extract `tenantId` → upload with key `${tenantId}/insurance-email-attachments/${ts}-${filename}`
3. If no match found → skip upload entirely (current behavior: uploads then discards - this is an improvement)
4. Pass `tenantId` and attachment keys together to `InsuranceEmailReplyService`


**Bonus:** Eliminates wasted S3 writes for unmatched emails.

**Staging prefix approach was rejected** - it would add two S3 operations per attachment (write + copy + delete), require an orphan cleanup job, and introduce more failure modes, all for no benefit since the lookup is synchronous and reliable.

### Video Consultation Recordings {#video-consultation-recordings}

**Model:** `VideoConsultation` has `tenantId` and `recordingUrls` (array of S3 URLs written by Tavus).

**Status:** Keys are **externally controlled** by Tavus. We cannot prefix keys with `tenantId` because Tavus determines the S3 key path. We only generate presigned GET URLs from existing keys.

**Mitigation: S3 object tagging.**

Since we can't control the key format, we use **S3 object tags** to associate recordings with tenants. This fills the gap for cost analysis, lifecycle management, and auditing without requiring a key rename.

**How it works:**

1. In `TavusEventHandlingService.handleConversationRecordingReady()`, after saving the recording URL, call `PutObjectTagging` on the S3 object:


```
await this.s3StorageService.tagObject({
 bucket: bucketName,
 key: s3_key,
 tags: { tenantId: consultation.tenantId },
});

```

1. One extra AWS API call per recording, only when the Tavus webhook fires (not per user request).


**Cost:**

- `PutObjectTagging` is a PUT request - **$0.005 per 1,000 requests**. Negligible given video consultation volume.
- No data transfer, no object copy - tagging is metadata-only.


**What tagging enables:**

- **S3 Storage Lens** - filter storage costs by `tenantId` tag
- **Lifecycle policies** - per-tenant retention rules (e.g., delete recordings after X days for specific tenants)
- **IAM tag-based policies** - future access control scoped by tenant tag
- **Compliance / audit** - query all objects belonging to a tenant


**Backfill for existing recordings:**

```
-- Extract S3 keys from existing VideoConsultation records
SELECT
 vc.id,
 vc."tenantId",
 unnest(vc."recordingUrls") AS recording_url
FROM "VideoConsultation" vc
WHERE array_length(vc."recordingUrls", 1) > 0;

```

For each row, extract the S3 key from the URL and call `PutObjectTagging`. Run as a one-time migration script.

**Limitations:**

- Tags don't change the key prefix - S3 bucket policies based on key prefix won't apply. Use IAM tag-based conditions instead.
- Max 10 tags per object (plenty of room).


**Priority:** Low - schedule after the main key-prefix migration is complete.

## Implementation Plan - New Code Changes {#implementation-plan-new-code-changes}

### Phase 1: Utility & Infrastructure {#phase-1-utility-infrastructure}

1. **Create a tenant-aware S3 key builder utility** to centralize key construction:


```
// s3-key-builder.util.ts

/**
 * Single-feature bucket: ${tenantId}/${...segments}
 */
function buildTenantScopedKey(tenantId: string, ...segments: string[]): string;

/**
 * Multi-feature bucket: ${tenantId}/${featurePrefix}/${...segments}
 */
function buildTenantFeatureKey(tenantId: string, featurePrefix: string, ...segments: string[]): string;

```

1. **Update **`SharedS3StorageService` to validate that every key starts with a UUID-shaped tenantId prefix. Log a warning for keys that don't match (to catch regressions during the transition period, without breaking existing reads).

### Phase 2: Fix Code Paths (organized by bucket/module) {#phase-2-fix-code-paths-organized-by-bucket-module}

For each affected file, the change follows this pattern:

1. Ensure `tenantId` is available in the method (either passed as parameter or obtained from service context)
2. Replace the current key construction with the key builder utility
3. Update any downstream code that reads the key from the database (read paths should work with both old and new key formats during migration)


**Work is organized by bucket/module** to minimize cross-developer conflicts. Each work unit is self-contained - one developer owns all changes for a given bucket, touching a predictable set of files. Work units with no file overlap can be developed in parallel on separate branches.

**Parallelization:** Work units A, B, C can run in parallel (no file overlap). D can run in parallel with E and F. E and F touch the same bucket but different modules - they can be parallelized if migration scripts are coordinated.

### Phase 3: Backward Compatibility {#phase-3-backward-compatibility}

During the transition period (old keys in DB, new keys being written):

- **Read paths** must handle both formats. The `getSignedUrlFromKey()` method already handles legacy URLs. No additional changes needed since reads use the key stored in the DB as-is.
- **Write paths** will use the new format immediately.
- **Delete paths** use the key from DB - will work with both old and new keys.

## Branding Keys Migration {#branding-keys-migration}

### Decision: Migrate from tenant.slug to tenantId {#decision-migrate-from-tenant-slug-to-tenantid}

**Status:** Approved - scheduled for Phase 2b

### Current State {#current-state}

Branding uploads use `tenant.slug` in the S3 key:

```
branding/${tenant.slug}/clinicLogo-${timestamp}.jpg (clinic portal)
branding/${tenant.slug}/${fileType}-${timestamp}.jpg (enterprise portal)

```

The S3 key is stored inside `tenant.settings` JSON (the legacy settings blob) under fields like `settings.mainLogo`, `settings.favicon`, `settings.enterpriseLogo`, `settings.consumerLogo`, `settings.referralLogo`, and `settings.branding.logo` (legacy clinic logo path).

### Why Migrate {#why-migrate}

1. **Slug collision between tenant types.** The uniqueness constraint is `@@unique([slug, type])`, meaning a CLINIC and CONSUMER tenant can (and do) share the same slug `"mars"`. Both portals upload branding to `branding/mars/`, resulting in files from different tenant IDs coexisting under the same prefix. This works by accident (filenames differ), but it violates tenant isolation.
2. **Slug mutability.** Nothing prevents a slug from being changed. If a tenant renames their slug, old branding files are orphaned under the old prefix, new uploads go to the new prefix, and there is no way to discover the orphaned files without manual investigation. `tenantId` (UUID) is immutable.
3. **Consistency.** Every other S3 path in the system will use `tenantId` after this migration. One outlier using slug adds cognitive overhead and special-case logic.
4. **Cost analysis.** Per-tenant S3 cost attribution via prefix-based tools (S3 Storage Lens, S3 Inventory) will not capture branding files unless they follow the same `${tenantId}/...` convention.

### Target Key Format {#target-key-format}

```
${tenantId}/branding/${fileType}-${timestamp}.${ext}

```

Examples:

- `a1b2c3d4-ef56-7890-abcd-ef1234567890/branding/clinicLogo-1737123456789.jpg`
- `a1b2c3d4-ef56-7890-abcd-ef1234567890/branding/mainLogo-1737123456789.png`

### Affected Files {#affected-files}

| File | Current Key Pattern | Change |
| --- | --- | --- |
| clinic/settings/settings.service.ts | branding/${tenant.slug}/clinicLogo-${ts}.${ext} | Replace tenant.slug with tenantId |
| enterprise/settings/settings.service.ts | branding/${tenant.slug}/${fileType}-${ts}.${ext} | Replace tenant.slug with tenantId |

### Migration Script for Existing Data {#migration-script-for-existing-data}

Branding keys are stored in the `tenant.settings` JSON column, which requires JSON field updates rather than simple column updates.

**Step 1: Extract all branding S3 keys from the settings JSON**

```
SELECT
 t.id AS tenant_id,
 t.slug,
 t.type,
 t.settings->>'mainLogo' AS main_logo_key,
 t.settings->>'favicon' AS favicon_key,
 t.settings->>'enterpriseLogo' AS enterprise_logo_key,
 t.settings->>'consumerLogo' AS consumer_logo_key,
 t.settings->>'referralLogo' AS referral_logo_key,
 t.settings->'branding'->>'logo' AS legacy_clinic_logo_key
FROM "Tenant" t
WHERE t.settings IS NOT NULL
 AND t.settings != 'null'::jsonb
 AND (
 t.settings->>'mainLogo' IS NOT NULL
 OR t.settings->>'favicon' IS NOT NULL
 OR t.settings->>'enterpriseLogo' IS NOT NULL
 OR t.settings->>'consumerLogo' IS NOT NULL
 OR t.settings->>'referralLogo' IS NOT NULL
 OR t.settings->'branding'->>'logo' IS NOT NULL
 );

```

**Step 2: For each non-null key, if it starts with **`branding/`**:**

1. S3 CopyObject from `branding/${slug}/${file}` to `${tenantId}/branding/${file}`
2. Update the JSON field with the new key:


```
-- Example: update mainLogo key
UPDATE "Tenant"
SET settings = jsonb_set(
 settings,
 '{mainLogo}',
 to_jsonb(CONCAT(id, '/branding/', SPLIT_PART(settings->>'mainLogo', '/', 3)))
)
WHERE id = ''
 AND settings->>'mainLogo' LIKE 'branding/%';

-- Example: update legacy nested clinic logo
UPDATE "Tenant"
SET settings = jsonb_set(
 settings,
 '{branding,logo}',
 to_jsonb(CONCAT(id, '/branding/', SPLIT_PART(settings->'branding'->>'logo', '/', 3)))
)
WHERE id = ''
 AND settings->'branding'->>'logo' LIKE 'branding/%';

```

**Step 3: Verify and log in migration audit table (same as other scripts)**

### Notes {#notes}

- The `resolveBrandingAssetUrl()` utility handles both full URLs and S3 keys transparently, so it will work with the new key format without changes.
- Test fixtures in `branding-test.constants.ts` should be updated to use `${tenantId}/branding/` prefixed keys.
- Branding files that are already full URLs (`http://...` or `https://...`) or absolute paths (`/...`) are skipped - the resolver already returns them as-is.

## Tavus Recording Keys {#tavus-recording-keys}

### Decision: No action - accept current state {#decision-no-action-accept-current-state}

**Status:** Decided - no migration needed

### Current State {#current-state}

Video consultation recordings are written to the `VIDEO_CONSULTATION_RECORDINGS` bucket by **Tavus** (third-party service), not by our application. The flow is:

1. We call the Tavus Create Conversation API with `enable_recording: true`, `recording_s3_bucket_name`, `recording_s3_bucket_region`, and `aws_assume_role_arn`.
2. Tavus assumes our IAM role and writes the recording to our S3 bucket using **its own key format** (e.g., `conversations/{conversation_id}/recording.mp4`).
3. Tavus sends an `application.recording_ready` webhook with the `s3_key`.
4. Our `TavusEventHandlingService` receives the webhook, constructs a full S3 URL, and appends it to `VideoConsultation.recordingUrls`.
5. When a user requests the recording, we extract the S3 key from the URL and generate a presigned GET URL.

### Why we cannot add tenantId to the key {#why-we-cannot-add-tenantid-to-the-key}

The Tavus Create Conversation API does **not** support a `recording_s3_key_prefix` or any parameter to customize the S3 key path. The only recording-related properties accepted are:

- `recording_s3_bucket_name`
- `recording_s3_bucket_region`
- `aws_assume_role_arn`


The S3 key is entirely controlled by Tavus internally. We have no way to inject a tenantId prefix into the key at write time.

### Why this is acceptable {#why-this-is-acceptable}

1. **Dedicated single-feature bucket.** `VIDEO_CONSULTATION_RECORDINGS` is only used for Tavus recordings. No other feature writes to it, so there is no cross-feature confusion.
2. **Tenant scoping via DB is fully functional.** The `VideoConsultation` model has a `tenantId` column. All queries for recordings are tenant-scoped through the database. Tenant offboarding can delete recordings by querying `VideoConsultation WHERE tenantId = ?` and removing the corresponding S3 objects.
3. **Low volume.** One recording per video consultation. This is not a high-throughput upload path.
4. **Post-copy is possible but not worth the cost.** We could copy each recording to a tenantId-prefixed key after receiving the webhook, update the DB, and delete the original. However, this adds latency, S3 transfer costs, and complexity for a marginal benefit.

### Future consideration {#future-consideration}

If Tavus adds a `recording_s3_key_prefix` parameter in the future, we can adopt it with a one-line change in `CommonTavusService.createConversation()`. This is worth raising with the Tavus account manager as a feature request, but should not block this migration.

## NULL petId Documents {#null-petid-documents}

### Decision: Skip during migration, log for manual review {#decision-skip-during-migration-log-for-manual-review}

**Status:** Decided

### Data {#data}

Confirmed from database dumps across environments:

| Environment | Count | Source |
| --- | --- | --- |
| Prestaging | 3 | CONSUMER_UPLOAD |
| Staging | 12 | CONSUMER_UPLOAD |
| Production | 21 | CONSUMER_UPLOAD |

All records are `CONSUMER_UPLOAD` source (stored in the `DOCUMENTS` bucket). The increasing count across environments suggests these are real user uploads where the pet association was lost or never set, not just test data.

### Why they can't be auto-migrated {#why-they-can-t-be-auto-migrated}

The `Document` model has no `tenantId`. The only path to a tenant is `petId` → `Pet` → `PetTenant`. With `petId = NULL`, there is no way to programmatically resolve a tenantId.

### Why this is not a concern {#why-this-is-not-a-concern}

- No production code path creates documents without `petId` - both `consumer/documents.service.ts` and `clinic/clinic-documents.service.ts` validate pet ownership before creating a document.
- The consumer documents service already throws `ClientNotFoundError("Document has no associated pet")` when encountering NULL petId, meaning these records are effectively inaccessible through the API.
- 21 records in production is still a very small number relative to total documents, but the pattern across environments (3 → 12 → 21) suggests a code path may have allowed this in the past. Worth investigating the root cause separately to prevent future occurrences.

### Handling strategy {#handling-strategy}

1. **Migration script:** Skip these records (21 in production). Log them to the migration audit table with status `skipped_no_tenant`:


```
INSERT INTO _s3_migration_audit (table_name, record_id, bucket, old_key, new_key, status, error_message)
SELECT
 'Document',
 d.id,
 'DOCUMENTS',
 d."documentPath",
 '',
 'skipped_no_tenant',
 'petId is NULL - cannot resolve tenantId'
FROM "Document" d
WHERE d."petId" IS NULL AND d."deletedAt" IS NULL;

```

1. **Post-migration:** Manually review the 21 production records. For each one, either:Link to the correct pet if identifiable from `documentName` or `documentPath`
2. Soft-delete (`SET deletedAt = NOW()`) if they are orphaned legacy data
3. **S3 objects:** Leave in place under the old key. They remain accessible if anyone somehow references them directly.
