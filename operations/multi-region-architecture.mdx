# 🟢 RECOMMENDED MODEL: ACTIVE-ACTIVE WITH EKS AUTO MODE

## 1️⃣ Executive Summary

The recommended enterprise architecture is **Active-Active multi-region deployment enhanced with EKS Auto Mode**.

Both Singapore and US East operate as fully active production regions.

By enabling **EKS Auto Mode**, infrastructure management is automated, reducing operational overhead while maintaining high scalability and resilience.

### Strategic Benefits

✔ Near-zero downtime
✔ Global performance optimization
✔ Automated infrastructure scaling
✔ Reduced DevOps complexity
✔ Enterprise-grade resilience

## 2️⃣ Why EKS Auto Mode Strengthens Active-Active

In a traditional Active-Active setup, each region requires:

- Node group sizing
- Capacity forecasting
- Patch management
- Scaling policy tuning


With **EKS Auto Mode**, AWS manages:

- Compute provisioning
- Node lifecycle
- Scaling decisions
- Infrastructure right-sizing


This ensures:

- Faster scaling during traffic spikes
- Faster recovery during regional failover
- Lower operational risk
- Reduced infrastructure misconfiguration

## 3️⃣ Architecture Overview

### Global Flow

Users Worldwide
↓
AWS Global Accelerator
↓
Singapore (Active)
+
US East (Active)

Traffic is distributed using latency-based and health-based routing.

## 4️⃣ Regional Architecture (Per Region)

Each region includes:

### 🌐 Network Layer

- 3 Availability Zones
- Public Subnets (ALB)
- Private Subnets (EKS, DB, Cache)
- NAT per AZ

### ☸️ Application Layer (EKS Auto Mode Enabled)

Using:

- Amazon EKS (Auto Mode)


Capabilities:

- Automatic node provisioning
- Automatic horizontal scaling
- Pod rescheduling across AZs
- Infrastructure patching handled by AWS
- Self-healing cluster behavior


Impact During Regional Failover:

If one region fails:

- Traffic shifts
- Remaining region auto-scales nodes automatically
- No manual node provisioning required


This significantly improves resilience under sudden traffic spikes.

### 🗄 Database Layer

Using:

- Amazon Aurora Global Database


Singapore:

- Primary writer (Multi-AZ)


US East:

- Secondary cluster (read optimized)


Features:

- Sub-second replication
- Automatic storage scaling
- Fast cross-region promotion


RPO: < 1 second
RTO: Near zero

### 🔴 Cache Layer

Using:

- Amazon ElastiCache Global Datastore


Features:

- Multi-AZ Redis
- Cross-region replication
- Automatic failover

## 5️⃣ High Availability Strategy

### AZ Failure

- Pods rescheduled automatically
- DB fails over within region
- Redis promotes replica
- No downtime

### Region Failure

- Global Accelerator reroutes traffic
- Remaining region auto-scales via EKS Auto Mode
- Aurora promoted if required
- Redis replica promoted


Downtime: Minimal to near-zero

## 6️⃣ RTO / RPO Targets

| Metric | Target |
| --- | --- |
| RTO | Near Zero |
| RPO | < 1 Second |
| Availability | 99.99%+ |

## 7️⃣ Operational Impact

With EKS Auto Mode:

- No node group management
- No AMI maintenance
- Reduced DevOps workload
- Faster scaling during peak demand
- Simplified multi-region management


This lowers operational risk in a complex Active-Active environment.

## 8️⃣ Cost Consideration

Active-Active + EKS Auto Mode:

Higher infrastructure cost due to:

- Fully active clusters in both regions
- Full database and cache capacity


However:

✔ Cost justified by uptime
✔ Reduced operational overhead
✔ Improved scalability
✔ Lower human-error risk

# 🔵 ALTERNATIVE MODEL: ACTIVE-PASSIVE WITH EKS AUTO MODE

## 1️⃣ Executive Summary

Active-Passive operates Singapore as production and US East as standby.

EKS Auto Mode still provides benefits by:

- Keeping standby region warm
- Scaling quickly if failover occurs
- Reducing infrastructure management

## 2️⃣ Architecture Flow

Users
↓
AWS Global Accelerator
↓
Singapore (Active)
↕ Replication
US East (Standby)

## 3️⃣ Regional Design

### Singapore

- Fully scaled EKS Auto Mode cluster
- Aurora Primary
- Redis Primary

### US East

- Minimal baseline EKS Auto Mode capacity
- Aurora Secondary
- Redis Replica


During failover:

- Traffic shifts
- Auto Mode scales nodes automatically

## 4️⃣ RTO / RPO

| Metric | Target |
| --- | --- |
| RTO | < 5 Minutes |
| RPO | < 1 Second |
| Availability | 99.9-99.99% |

## 5️⃣ Cost Profile

Lower cost because:

- Standby region runs minimal compute
- Auto Mode scales only when needed


Estimated 30-40% lower than Active-Active.
