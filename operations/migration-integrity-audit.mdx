## How to Fix a Modified Migration

When a migration file has been modified after it was applied, the correct fix is a **two-step revert + new migration** approach:

### Step 1: Revert the migration file to its original (applied) state

Restore the `migration.sql` to exactly what was applied on prestaging. This makes the file checksum match the `_prisma_migrations` record again.

```
# Find the commit where the migration first landed on prestaging
git log origin/prestaging --format='%H %s' --follow --  | tail -1

# Restore the file to that commit's version
git checkout  --
```

After this, the file on disk matches what the DB thinks was executed ,  **checksum restored, Prisma is happy**.

### Step 2: Create a new migration for the intended changes

The changes that were incorrectly added to the old migration need to go into a **new, properly sequenced migration**. Use the standard Prisma workflow:

```
# If the change is a schema change (new column, table, etc.)
# 1. Update the Prisma schema file(s)
# 2. Generate a new migration
pnpm prisma:migration:create

# If the change is non-schema (enum values, grants, functions, etc.)
# Create a new migration with --create-only, then add the SQL
pnpm prisma:migration:create
# Edit the NEW migration file with the needed SQL
```

### Why this works

- **Prestaging/production:** The old migration was already applied ,  reverting the file just fixes the checksum. The new migration applies the missing changes as a separate, tracked step.
- **Fresh environments:** The old migration runs with its original SQL, then the new migration applies the additional changes. End state is identical to prestaging.
- **No drift:** All environments converge to the same final DB state through the same ordered sequence of migrations.

### Reference: PW-4593 (PR #3766)

This is exactly what was done for violation #4 (`WELLNESS_PLAN_REMINDER`). The migration file was reverted to its original state, restoring the checksum. Since the enum value already existed on prestaging (it was in the original migration that was applied), no follow-up migration was needed in that case.

### Violation #3 (add_role_templates) ,  High risk

4 `AuditLogActionType` enum values were added to an already-applied migration.

**Fix:**

1. Revert `migration.sql` to the version at first merge (`324dc33e2`) ,  without the enum `ALTER TYPE` statements
2. Create a new migration (e.g. `pnpm prisma:migration:create add_role_template_audit_log_types`) containing the 4 `ALTER TYPE` statements
3. Verify on prestaging: if the enum values already exist (applied manually or via another path), the new migration's `ADD VALUE` will fail with "already exists" ,  in that case, use `ADD VALUE IF NOT EXISTS` (PostgreSQL 9.3+) to make it idempotent

**Fix:**

1. Revert `migration.sql` to the version at first merge (`2d6145054`) ,  with the hardcoded `FOR ROLE postgres`
