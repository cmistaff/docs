## Executive Summary

This document outlines a feature enhancement to allow **automatic confirmation of appointment procedures** based on payment conditions, while providing clinics with granular control over payment timing and confirmation rules.

### Key Capabilities

- **Auto-confirm procedures** based on: instant booking, full payment, partial payment, or deposit
- **Flexible payment timing**: immediate at booking, after confirmation, on-site only, or post-service
- **Clinic-level soft lock settings**: configurable hold duration for unpaid bookings
- **Backward compatible**: existing behavior preserved as default

---

## Problem Statement

### Current State

```
Consumer Books -> AppointmentStep(status=PENDING)
                        |
         Clinic CONFIRMS manually
                        |
        AppointmentStep(status=CONFIRMED)
                        |
         Consumer CAN NOW pay
                        |
        AppointmentStep(paymentStatus=PAID)
```

**The Blocking Code** (`server/src/modules/portals/shared/payments/services/core-payment.service.ts`):

```typescript
appointmentSteps: {
  where: {
    serviceStepId: { in: serviceStepIds },
    paymentStatus: AppointmentStepPaymentStatus.UNPAID,
    status: AppointmentStatus.CONFIRMED,  // <-- BLOCKER
  },
}
```

### The Chicken-and-Egg Problem

1. Payment **requires** `CONFIRMED` status
2. Auto-confirm based on payment **cannot happen** because payment requires confirmation first
3. This circular dependency prevents pay-to-confirm workflows

### Business Impact

- Clinics cannot offer "pay now to secure your spot" booking flows
- No option for instant confirmation of low-risk procedures
- Limited flexibility for different procedure types (free, deposit-based, full-payment)
- Consumer experience friction: must wait for manual clinic confirmation

## Proposed Solution

### Core Concept: Decouple Payment Enablement from Confirmation

Instead of coupling payment to `CONFIRMED` status, introduce:

1. **`PaymentTiming`** - When is payment enabled/allowed?
2. **`AutoConfirmRule`** - When should auto-confirmation trigger?
3. **`isPaymentEnabled`** - Instance-level flag on AppointmentStep
4. **Clinic-level soft lock settings** - How long to hold unpaid slots?

### Solution Overview

```
+-------------------------------------------------------------+
|                    CONFIGURATION LAYER                        |
|  (ServiceStep - Procedure Template)                          |
+-------------------------------------------------------------+
|  paymentTiming: IMMEDIATE | AFTER_CONFIRM | ON_SITE | POST  |
|  autoConfirmRule: MANUAL | INSTANT | ON_FULL | ON_PARTIAL    |
|  autoConfirmThreshold: number (for partial payment %)        |
+-------------------------------------------------------------+
                              |
                              v
+-------------------------------------------------------------+
|                    INSTANCE LAYER                             |
|  (AppointmentStep - Booked Instance)                         |
+-------------------------------------------------------------+
|  isPaymentEnabled: boolean                                   |
|  paymentEnabledAt: DateTime                                  |
|  confirmedBy: 'manual' | 'auto_instant' | 'auto_payment'    |
+-------------------------------------------------------------+
                              |
                              v
+-------------------------------------------------------------+
|                    CLINIC SETTINGS                            |
|  (ClinicBookingSettings - Tenant Level)                      |
+-------------------------------------------------------------+
|  softLockDurationMinutes: number                             |
|  autoCancelOnSoftLockExpiry: boolean                         |
|  sendReminderBeforeExpiry: boolean                           |
+-------------------------------------------------------------+
```

## Technical Design

### New Enums

```prisma
// Payment timing - when is online payment allowed?
enum PaymentTiming {
  IMMEDIATE_AT_BOOKING    // Payment enabled right when consumer books
  AFTER_CONFIRMATION      // Current behavior - wait for clinic to confirm (DEFAULT)
  ON_SITE_ONLY            // No online payment - clinic collects in person
  POST_SERVICE            // Pay after the service is rendered
}

// Auto-confirmation rules - when should system auto-confirm?
enum AutoConfirmRule {
  MANUAL                  // Current behavior - clinic confirms manually (DEFAULT)
  INSTANT                 // Auto-confirm immediately upon booking
  ON_FULL_PAYMENT         // Auto-confirm when 100% paid
  ON_PARTIAL_PAYMENT      // Auto-confirm when X% paid (uses autoConfirmThreshold)
  ON_DEPOSIT              // Auto-confirm when deposit paid
}
```

### Configuration Matrix

| PaymentTiming | AutoConfirmRule | Behavior |
|---------------|-----------------|----------|
| `AFTER_CONFIRMATION` | `MANUAL` | **Current default** - clinic confirms, then consumer pays |
| `IMMEDIATE_AT_BOOKING` | `ON_FULL_PAYMENT` | Pay-to-confirm: full payment required |
| `IMMEDIATE_AT_BOOKING` | `ON_DEPOSIT` | Pay-to-confirm: deposit required |
| `IMMEDIATE_AT_BOOKING` | `ON_PARTIAL_PAYMENT` | Pay-to-confirm: X% required |
| `ON_SITE_ONLY` | `INSTANT` | Free/trusted procedures - instant confirm, pay at clinic |
| `ON_SITE_ONLY` | `MANUAL` | Clinic confirms, payment collected on-site |
| `POST_SERVICE` | `MANUAL` | Pay after appointment (invoicing flow) |

## Edge Cases

| # | Scenario | Expected Behavior |
|---|----------|-------------------|
| 1 | Consumer pays partially, below threshold, soft lock expires | Auto-cancel if enabled; partial payment refunded or held as credit |
| 2 | Consumer pays after soft lock expires (not cancelled) | Payment accepted, auto-confirm rules still apply |
| 3 | Consumer pays after appointment auto-cancelled | Error: "Appointment no longer available, please rebook" |
| 4 | Clinic manually confirms before consumer pays (pay-to-confirm) | Override auto-confirm; status=CONFIRMED, consumer can still pay |
| 5 | Consumer cancels during soft lock with partial payment | Normal cancellation; refund per cancellation policy |
| 6 | Multiple steps with different auto-confirm rules | Each step evaluated independently |
| 7 | Free procedure ($0 price) | Auto-set paymentStatus=PAID, apply autoConfirmRule |
| 8 | Network/referral appointment | Apply clinic's autoConfirmRule; commission still tracked |
| 9 | Soft lock expires during active payment (card processing) | Grace period applies; extend lock during payment attempt |
| 10 | Consumer pays exactly threshold amount | Confirm (threshold is minimum, not exclusive) |

## Migration Strategy

### Phase 1: Schema Migration (Non-Breaking)

1. Add new columns with defaults matching current behavior:
   - `ServiceStep.paymentTiming` = `AFTER_CONFIRMATION`
   - `ServiceStep.autoConfirmRule` = `MANUAL`
   - `AppointmentStep.isPaymentEnabled` = `false`
   - `AppointmentStep.confirmedBy` = `null`

2. Create `ClinicBookingSettings` table with sensible defaults

3. **Data migration**: For existing CONFIRMED appointments:
   ```sql
   UPDATE "AppointmentStep"
   SET "isPaymentEnabled" = true,
       "confirmedBy" = 'manual'
   WHERE "status" = 'CONFIRMED';
   ```

### Phase 2: Backend Logic Updates

1. Update `CorePaymentService` to check `isPaymentEnabled` instead of `CONFIRMED`
2. Add auto-confirm logic to `PaymentCompletionService`
3. Add booking flow logic to set `isPaymentEnabled` based on `ServiceStep.paymentTiming`
4. Implement soft lock expiry BullMQ jobs

### Phase 3: API & Frontend Updates

1. Update ServiceStep DTOs and endpoints
2. Add ClinicBookingSettings endpoints
3. Update consumer booking flow UI
4. Update clinic procedure configuration UI
5. Update clinic settings UI

### Phase 4: Rollout

1. Feature flag for gradual rollout
2. Enable for beta clinics first
3. Monitor metrics
4. Full rollout
